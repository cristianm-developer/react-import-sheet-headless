# 8. Edit — Plan de construcción

## Objetivo

Construir el módulo **Edit** (headless) que permita al consumidor del provider obtener el **resultado final** (sheet validada y transformada) junto con los **errores existentes**, y **editar valores** de celdas de forma granular. Tras cada edición se ejecuta la misma lógica de validación y transformación pero **solo en el ámbito afectado**: la celda editada, la fila que la contiene y la hoja (table). Todo sin UI: solo datos, hooks y provider.

---

## Entrada y salida

| Concepto | Origen | Descripción |
|----------|--------|-------------|
| **Entrada** | Provider (Setting) + Transform (7. Transform) | **sheet** (output del transform: mismo esquema con `name`, `sheetLayout`, `errors`, `rows` → `row` con `index`, `errors`, `cells` → **cell** con `key`, `value`, `errors`). Es la sheet que el provider ya expone como resultado validado y transformado. |
| **Salida** | Mismo esquema (tipos del Setting) | **sheet** actualizada: mismo contrato; solo cambian los valores y/o errores de la celda editada, de su fila y de la hoja según corresponda tras re-ejecutar validadores y transforms en el ámbito indicado. |

El provider retornará el **resultado** (sheet final) junto con los **errores** (ya presentes en `sheet.errors`, `row.errors`, `cell.errors`) y una forma de **editar** que dispara la pipeline de validación + transformación acotada.

Además, se expone un **resultado paginado** para que la UI (fuera de este módulo) pueda renderizar y editar los datos por páginas: `{ page, pageSize, totalCount, rows }`, donde **rows** mantiene el mismo esquema que ya conocemos (cada `row` con `index`, `errors`, `cells` → **cell** con `key`, `value`, `errors`).

---

## Resultado paginado (estado derivado, verdad única)

Para permitir a la UI consumir y mostrar el resultado por páginas de forma **reactiva** (sin llamar funciones manualmente en cada render), el hook **useSheetEdit** recibe `page` y `pageSize` y devuelve **estado derivado**:

```ts
interface PaginatedResult<TRow> {
  page: number;       // página actual (1-based o 0-based, según convención)
  pageSize: number;   // tamaño de página solicitado
  totalCount: number; // total de filas en la sheet (sheet.rows.length)
  totalPages: number; // total de páginas (derivado)
  rows: TRow[];       // filas de la página actual; mismo esquema que row (index, errors, cells)
}

// useSheetEdit recibe page + pageSize y devuelve estado derivado (no una función a llamar)
const { pageData, totalPages, editCell, sheet, ... } = useSheetEdit({ page, pageSize });
// pageData: PaginatedResult (page, pageSize, totalCount, totalPages, rows)
```

- **pageData**: resultado paginado calculado a partir de la **sheet** actual; al cambiar la sheet o `page`/`pageSize`, React re-renderiza y el consumidor siempre tiene el slice correcto.
- **rows**: subconjunto de `sheet.rows` correspondiente a la página; cada elemento es la **misma referencia** que en la sheet (inmutabilidad estructural), para que React.memo en filas no re-renderice filas no afectadas.
- **Índice global**: las filas en `pageData.rows` conservan su `row.index` (o posición global en la sheet). La UI debe usar siempre ese **rowIndex global** en **editCell**, nunca el índice local dentro de la página, para que la pipeline y el estado no se confundan.

## Contrato de edición: `editCell`

Se expone una interfaz para editar una celda por **posición** y **nuevo valor**:

```ts
interface EditCellParams {
  rowIndex: number;  // índice de la fila (row.index)
  cellKey: string;   // key del campo (cell.key, coincide con el field del layout)
  value: unknown;    // nuevo valor a asignar a la celda
}

// Ejemplo de uso desde el hook
editCell({ rowIndex: 2, cellKey: 'email', value: 'nuevo@mail.com' });
```

- **rowIndex**: identifica la fila en `sheet.rows` (por `row.index` o por posición en el array, según cómo se normalice en el provider).
- **cellKey**: identifica la celda dentro de la fila (coincide con la clave del **field** en el `sheetLayout`).
- **value**: valor nuevo que reemplazará a `cell.value` antes de re-ejecutar validación y transformación.

Tras actualizar el valor, se ejecuta la **pipeline de re-validación y re-transformación** solo para el ámbito afectado (ver más abajo).

---

## Pipeline tras `editCell` (orden de ejecución)

Tras aplicar el nuevo valor a la celda indicada, se ejecuta **en secuencia** y de forma **async** (misma filosofía que Validator y Transform, pero acotado):

1. **Cell validator** (solo esa celda)  
   - Obtener la fila por `rowIndex`, la celda por `cellKey`.  
   - Ejecutar en secuencia los **cell-validators** del field correspondiente (`sheetLayout.fields[cellKey].validators`).  
   - Actualizar **solo** `cell.errors` de esa celda.

2. **Row validator** (solo esa fila)  
   - Ejecutar en secuencia los **row-validators** del layout para **esa fila únicamente**.  
   - Actualizar **solo** `row.errors` de esa fila.

3. **Table (sheet) validator** (toda la hoja) — **opcional / lazy**  
   - Ejecutar solo si el layout lo indica para ese campo (p. ej. bandera `triggersSheetValidation`) o si la hoja está marcada como "dirty". Así se evita ejecutar validadores pesados (ej. búsqueda de duplicados en 50k filas) en cada pulsación.  
   - Ejecutar en secuencia los **sheet-validators** del layout sobre la **sheet completa** (ya con la celda y la fila actualizadas).  
   - Actualizar `sheet.errors`.

4. **Transforms** (misma lógica que 7. Transform, pero solo en el ámbito afectado):
   - **Reglas** (igual que 7. Transform):
     - Si `sheet.errors.length > 0` tras validación → no ejecutar **ningún** transform (cell, row, sheet).
     - Si `row.errors.length > 0` (de esa fila) → no ejecutar transforms de **esa fila** (ni cell ni row).
     - Si `cell.errors.length > 0` (de esa celda) → no ejecutar transform de **esa celda**.
   - **Orden**:
     - **Cell transforms** (solo esa celda): si la celda no tiene errores, ejecutar en secuencia las `field.transformations` para esa celda; actualizar el valor de la celda.
     - **Row transforms** (solo esa fila): si la fila no tiene errores, ejecutar en secuencia los **row-transformations** del layout para **esa fila únicamente**.
     - **Sheet transforms**: si la hoja no tiene errores, ejecutar en secuencia los **sheet-transformations** sobre la sheet completa.

Resumen: **actualizar valor → cell validators (esa cell) → row validators (esa row) → sheet validators → cell transforms (esa cell, si no cell/row/sheet errors) → row transforms (esa row) → sheet transforms.** El resultado es una **nueva sheet** (inmutable) con el mismo esquema, actualizada solo donde corresponda, que se escribe en el **provider**.

---

## Estado en el provider

- El provider expone:
  - **sheet** (o nombre equivalente): el resultado final validado y transformado, con `errors` en hoja, filas y celdas.
  - **editCell**: función que acepta `EditCellParams` y envía la edición al **Worker** (vía Comlink); el Worker ejecuta la pipeline y devuelve la nueva sheet; el provider actualiza el estado. Así se evita que un sheet-validator pesado (ej. 50k filas) bloquee el main thread y la UI mantenga 60fps.
  - **Paginación como estado derivado**: el hook **useSheetEdit** recibe `{ page, pageSize }` y devuelve **pageData** (y **totalPages**) calculados a partir de la sheet actual, de forma reactiva; no se expone una función `getPaginatedResult` que el desarrollador deba llamar en cada render.
- Tras cada `editCell` el estado del provider se actualiza con la **sheet** resultante (inmutabilidad estructural: solo cambian la fila editada y la referencia de la sheet; el resto de filas conservan identidad). Los consumidores reaccionan a ese estado; la paginación se deriva de la sheet actual.
- **Worker**: la pipeline de edición (runEditPipeline) se ejecuta en el **Worker** (Comlink), igual que Validator/Transform, para consistencia y para garantizar que la UI no se congele aunque el sheet-validator sea costoso.

---

## Hook de edición (headless)

- **useSheetEdit(options)**: hook que consume el contexto del provider. Opciones sugeridas: `{ page, pageSize, debounceMs?, submitOnBlur? }`. Devuelve:
  - **sheet**: la sheet actual (resultado + errores).
  - **editCell(params: EditCellParams)**: función que encola la edición; la pipeline se ejecuta en el **Worker** (vía Comlink). Opcionalmente se aplica **debounce** (ver más abajo).
  - **pageData**: estado derivado `PaginatedResult` (`page`, `pageSize`, `totalCount`, `totalPages`, `rows`) calculado a partir de la sheet actual y de `page`/`pageSize`; la UI no llama a ninguna función, solo usa `pageData.rows` y `pageData.totalPages`.
  - **totalPages**: derivado de `totalCount` y `pageSize` (conveniencia).
  - Opcional: **status** del último edit (`idle | editing | success | error`).

**Debounce (recomendado)**:
- El hook debe ofrecer o recomendar **debounce**: no disparar la pipeline en cada pulsación de tecla. Opciones:
  - **submitOnBlur**: disparar la pipeline al salir del campo (`onBlur`); el valor “optimista” puede mostrarse en UI mientras tanto.
  - **debounceMs**: disparar la pipeline solo cuando el usuario deje de escribir durante X ms.
- Así se evita ejecutar validación/transform de hoja en cada letra al corregir un nombre largo.

El hook (y el Worker) deben:
- Resolver la fila por `rowIndex` (índice **global** en la sheet).
- Resolver la celda por `cellKey` dentro de esa fila.
- Construir una **copia estructuralmente inmutable** de la sheet: **solo la fila afectada** y la referencia de la sheet cambian; las filas `j !== rowIndex` conservan la **misma referencia** en memoria. Esto permite que React.memo en filas evite re-renders de filas no tocadas.
- Invocar en el Worker los runners de validación y transformación solo para el ámbito afectado; sheet-validator solo si el layout lo indica o la hoja está "dirty".

---

## Reutilización de Validator y Transform

- **Validación**: reutilizar `runCellValidators(cell, field, sheetLayout)` para esa celda; `runRowValidators(row, sheetLayout)` para esa fila; `runSheetValidators(sheet, sheetLayout)` para la hoja. No es necesario re-ejecutar validación en todas las filas/celdas.
- **Transformación**: reutilizar `runCellTransforms(cell, field, sheetLayout)` para esa celda; `runRowTransforms(row, sheetLayout)` para esa fila; `runSheetTransforms(sheet, sheetLayout)` para la hoja. Las condiciones (no aplicar si hay errores en ese nivel) son las mismas que en 7. Transform.

Las implementaciones de validators y transforms siguen viviendo en **`utils/controller/[context]/`** (p. ej. `cell-email-validator.ts`, `cell-email-transform.ts`); los runners del Validator y del Transform las resuelven por nombre desde el Registry (el mismo que se alimenta con **Register()** en el layout o con registro manual). La diferencia con el flujo completo es solo de **alcance**: se ejecuta una sola celda, una sola fila y una vez la hoja, en lugar de iterar todas las filas/celdas.

---

## Dónde corre la pipeline (Worker, no main thread)

- **Decisión**: la pipeline **runEditPipeline** se ejecuta siempre en un **Web Worker** (vía Comlink), igual que el resto de pasos (Validator, Transform).
- **Motivo**: aunque validar una celda es ligero, el **sheet-validator** y el **sheet-transform** pueden ser pesados (ej. buscar duplicados en 50k filas). Si el usuario escribe rápido, ejecutar eso en el main thread laguearía la UI. Comlink hace que enviar la edición al Worker sea casi instantáneo y se mantienen 60fps.
- **Implementación**: exponer en el worker una función que reciba (sheet, sheetLayout, rowIndex, cellKey, value) y devuelva la nueva sheet; en el main, el hook llama a esa función vía Comlink y actualiza el estado del provider con el resultado.
- **Abort y ciclo de vida:** Si la edición usa el mismo Worker que Validator/Transform (o uno dedicado), ese Worker debe ser el mismo que se termina con **`abort()`** y en el cleanup al desmontar. No dejar un Worker de edición vivo si el usuario sale de la página. Ver **2. Setting** — *Abort y ciclo de vida*.

---

## Reglas de negocio (resumen)

- **Identificación**: edición por `rowIndex` (global) + `cellKey` + `value`.
- **Orden**: actualizar valor → cell validators → row validators → (opcional) sheet validators → cell/row/sheet transforms según reglas de errores.
- **Inmutabilidad estructural**: la sheet es inmutable en sentido de referencias: cada `editCell` produce una **nueva** referencia de **sheet** y una **nueva** referencia **solo de la fila editada**; las filas `j !== rowIndex` conservan la **misma referencia**. Así el Garbage Collector no sufre por clonar miles de filas en cada tecla, y React puede evitar re-renders en filas no afectadas (con React.memo).
- **Validación de hoja lazy**: ejecutar sheet-validator (y sheet-transform de hoja) solo cuando el layout lo indique (p. ej. campo con `triggersSheetValidation`) o cuando la hoja esté marcada como "dirty", para ahorrar ciclos cuando un cambio de celda no afecta reglas globales.
- **Errores**: se recalculan solo para la celda editada, su fila y (si aplica) la hoja; el resto de filas/celdas mantienen sus referencias y errores previos.

## Plan de tareas (orden sugerido)

### Fase A — Tipos y contrato

1. En `src/types` definir **EditCellParams** (`rowIndex`, `cellKey`, `value`) y, si aplica, un tipo para el resultado de la operación de edición (p. ej. `EditResult` con la sheet actualizada o error).
2. En `src/types` definir **PaginatedResult\<TRow\>** con `page`, `pageSize`, `totalCount`, `totalPages`, `rows` (rows con el mismo esquema que las filas de la sheet: index, errors, cells).
3. Asegurar que el **provider** (Setting) expone (o tendrá espacio para) la **sheet** final (validada + transformada) y que el tipo coincide con el esquema de 2. Setting.

### Fase B — Resolución de celda y actualización inmutable (shallow copy / inmutabilidad estructural)

3. Implementar **getRowByIndex(sheet, rowIndex)** que devuelva la fila correspondiente (por `row.index` o por índice en `rows`).
4. Implementar **getCellByKey(row, cellKey)** que devuelva la celda en la fila con ese `key`.
5. Implementar **setCellValue(sheet, rowIndex, cellKey, value)** que devuelva una **nueva sheet** con **inmutabilidad estructural**:
   - **Solo la fila afectada** (`rowIndex`) debe ser una **nueva** referencia (nueva fila con la celda actualizada).
   - Las filas `j !== rowIndex` se reutilizan **sin clonar**: misma referencia en el nuevo array de `sheet.rows`.
   - La referencia del objeto `sheet` es nueva; `sheet.rows` es un nuevo array (solo la posición `rowIndex` tiene una fila nueva).
   - Objetivo: que un cambio en la fila 5 no provoque re-render de la fila 6 cuando el desarrollador use React.memo en las filas de la tabla.

### Fase C — Pipeline de validación y transformación acotada

6. Implementar **runEditPipeline(sheet, sheetLayout, rowIndex, cellKey, value)** (o nombre equivalente) que:
   - Construya la sheet con el nuevo valor en la celda (usando setCellValue).
   - Ejecute runCellValidators para esa celda y actualice esa celda en la sheet.
   - Ejecute runRowValidators para esa fila y actualice esa fila en la sheet.
   - Ejecute runSheetValidators y actualice sheet.errors.
   - Aplique runCellTransforms (solo esa celda), runRowTransforms (solo esa fila), runSheetTransforms (sheet), respetando las reglas de “no transformar si hay errores” del Transform.
   - Devuelva la nueva sheet (inmutable).
7. Asegurar que los runners existentes (validator/transform) acepten una **fila** o **celda** suelta y la **sheet** para no re-validar/re-transformar todo el conjunto.

### Fase D — Resultado paginado (estado derivado e índice global)

9. Implementar el cálculo de **paginated result** como función pura **getPaginatedResult(sheet, page, pageSize)** que devuelva `{ page, pageSize, totalCount, totalPages, rows }` donde **rows** es el **slice** de `sheet.rows` para la página. Las filas en `rows` son las **mismas referencias** que en `sheet.rows` (no clonar), para preservar inmutabilidad estructural y que React.memo funcione. **Índice global**: cada fila en `rows` conserva su `row.index` (o posición global); la UI debe usar siempre ese valor en **editCell**, nunca el índice local dentro de la página.
10. En **useSheetEdit**, recibir **page** y **pageSize** como opciones y exponer **pageData** (y **totalPages**) como **estado derivado** calculado a partir de la sheet actual del provider (llamando a getPaginatedResult(sheet, page, pageSize) en el render). Así la paginación es reactiva y el desarrollador no tiene que invocar funciones manualmente; las funciones de edición siempre usan el rowIndex global correcto.

### Fase E — Hook, debounce y provider

11. Implementar **useSheetEdit(options)** que:
    - Consuma el contexto del provider y lea la **sheet** actual.
    - Acepte **page**, **pageSize** y opcionalmente **debounceMs** o **submitOnBlur**.
    - Exponga **pageData** y **totalPages** como estado derivado (getPaginatedResult(sheet, page, pageSize) en cada render).
    - Exponga **editCell(params)** que envíe la edición al Worker (Comlink); opcionalmente aplicar **debounce** (por tiempo o por blur) para no disparar la pipeline en cada tecla.
    - Opcional: **status** de la última edición (idle | editing | success | error).
12. Integrar en el **provider**: estado con la sheet final; el hook actualiza ese estado cuando el Worker devuelve la nueva sheet. Asegurar que runEditPipeline corre en el Worker.

### Fase F — Integración y pruebas (Vitest)

13. **Tests unitarios con Vitest**: setCellValue — inmutabilidad **estructural** (solo la fila editada es nueva referencia; las demás filas son la misma referencia); getRowByIndex / getCellByKey con datos de prueba.
14. **Tests con Vitest** de **getPaginatedResult**: page, pageSize, totalCount, totalPages y rows como slice correcto; rows son las **mismas referencias** que en sheet.rows para ese rango; índices globales correctos; casos borde (última página parcial, pageSize > totalCount).
15. **Tests con Vitest** de runEditPipeline (en Worker o mock): orden de validación/transform; validación de hoja solo cuando aplica (lazy); que la sheet devuelta tiene inmutabilidad estructural.
16. **Test con Vitest** del hook: editCell actualiza el estado vía Worker; sheet y pageData coherentes; que un cambio en una fila no invalida referencias de otras filas (criterio de éxito para React.memo).

## Criterios de aceptación (resumen)

- [ ] El provider expone el **resultado** (sheet validada y transformada) con **errores** en hoja, filas y celdas.
- [ ] Interfaz **editCell({ rowIndex, cellKey, value })**: identificación por fila (índice global), campo y nuevo valor.
- [ ] Tras **editCell**: pipeline en **Worker** (Comlink): cell validator (esa celda) → row validator (esa fila) → (opcional) sheet validator → cell/row/sheet transforms; validación de hoja **lazy** cuando el layout lo indique o hoja "dirty". Reglas de “no transformar si hay errores” que en 7. Transform.
- [ ] **Inmutabilidad estructural**: solo la fila editada y la sheet cambian de referencia; el resto de filas conservan identidad. **Criterio de éxito React**: un cambio en la fila 5 no provoca re-render de la fila 6 si el desarrollador usa React.memo en la tabla.
- [ ] **Paginación como estado derivado**: useSheetEdit recibe `{ page, pageSize }` y devuelve **pageData** y **totalPages**; la UI usa siempre **rowIndex global** en editCell (nunca índice local de página).
- [ ] Hook **useSheetEdit** expone `sheet`, `editCell`, **pageData**, **totalPages**; opcional **debounce** (debounceMs o submitOnBlur) para no disparar la pipeline en cada tecla.
- [ ] Reutilización de runners de validator/transform con alcance acotado; runEditPipeline corre en el Worker.
- [ ] **Tests unitarios con Vitest** para resolve, immutable-update (referencias), getPaginatedResult, runEditPipeline y useSheetEdit; suite en verde.

---

## Criterio de éxito técnico (React Performance)

- **Referencias compartidas**: Un cambio en la fila 5 **no** debe provocar el re-renderizado de la fila 6 cuando el desarrollador usa **React.memo** en las filas de su tabla. La arquitectura de inmutabilidad estructural (solo la fila editada y la sheet cambian de referencia; el resto de filas conservan identidad) debe garantizar esto.
