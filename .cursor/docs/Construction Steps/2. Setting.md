# 2. Setting — Fundamento del proyecto

## Objetivo

Establecer la base del proyecto: un **provider** que centralice el estado de importación (archivo CSV/sheet, datos crudos y estado de la operación). Este será el fundamento sobre el que se construirán el parser y el formateo. Además, se definen los **tipos compartidos** que usarán todos los módulos internos del sistema.

## Tipos compartidos (`src/types/`)

Todos los tipos de uso común entre módulos (parser, validadores, transformaciones, UI) vivirán en **`src/types/`**. Estructura prevista:

### 1. Error (I18n-ready: code + params)

Tipo base para mensajes de error en validación y flujo. **No usar strings estáticos** para permitir internacionalización: la UI traduce el `code` (y opcionalmente `params`). Ver **Architecture.md** — *Maturity & resilience → I18n of errors*.

- **Error / SheetError**: `{ code: string, params?: Record<string, unknown>, level?: 'error' | 'warning' | 'fatal' | 'info', message?: string }`
  - **code**: identificador estable para la UI (ej. `'INVALID_EMAIL'`, `'MIN_LENGTH'`). La UI decide el texto traducido.
  - **params**: metadatos para interpolación (ej. `{ value, min, actual }`).
  - **level**: severidad.
  - **message**: opcional, solo como fallback para depuración o UI por defecto; no debe ser el mensaje localizado final.

### 2. BaseSheet y resultado del parser (evitar duplicación)

Para no duplicar la estructura de datos en dos interfaces distintas, se usa **Generics**. Una interfaz base **`BaseSheet<T>`** donde `T` es el tipo de fila; así, si mañana se añade un campo (p. ej. `author`) a la hoja, solo se actualiza en un sitio.

- **BaseSheet\<TRow\>**: contrato base con `name`, `filesize`, `rowsCount?`, `headersCount?`, `headers`, `rows: TRow[]`. Cualquier campo común de hoja (mimetype, extension, etc.) vive aquí.
- **RawSheetCell**: `{ key: string, value: unknown }`
- **RawSheetRow**: `{ index: number, cells: RawSheetCell[] }`
- **RawSheet**: `BaseSheet<RawSheetRow>` — salida unificada del parser (CSV/XLSX/ODS) antes de validaciones ni layout. Incluye `name`, `filesize`, `rowsCount?`, `headersCount?`, `headers`, `rows`.

### 3. Sheet validada (sistema general)

Estructura que usa el sistema una vez aplicado layout y validaciones: hoja con errores por hoja, fila y celda. **No se duplica la forma de la hoja**: se reutiliza la base.

- **ValidatedRow**: `{ index: number, errors: Error[], cells: ValidatedCell[] }`
- **ValidatedCell**: `{ key: string, value: unknown, errors: Error[] }`
- **Sheet**: `BaseSheet<ValidatedRow>` extendido con metadatos de validación: `sheetLayout` (referencia a la plantilla), `mimetype`, `extension`, `errors: Error[]` (errores a nivel de hoja). Las filas son `ValidatedRow[]`.

### 4. SheetLayout (plantillas base, con versionado y registro por identificadores)

Define la plantilla contra la que se valida y transforma una hoja. Incluye **version** para compatibilidad con persistencia y migraciones (ver **Architecture.md** — *Schema versioning*; **11. Telemetry** — persistencia IndexedDB).

**Registro agnóstico (Zero-Bundle-Size):** El layout **no contiene funciones**; solo **identificadores (strings)**. Las funciones (validadores, sanitizers, transformadores) se registran por separado en el Provider. Esto permite **Tree Shaking perfecto**: si el usuario no registra el validador de email, ese código nunca llega al bundle del navegador. Ver *Registro agnóstico de validadores/sanitizers/transformadores* más abajo.

- **sheetLayout**:  
  - `name: string`  
  - **`version`**: `string | number` — versión del esquema del layout; se guarda con el estado persistido para detectar incompatibilidades y migrar.  
  - `description?: string`  
  - **sheet-validators** `string[]`: **identificadores** de validadores a nivel hoja (ej. `['unique-rows', 'max-size']`). El motor resuelve desde el Registry y ejecuta; devuelven `SheetError[]` (code + params, no strings).  
  - **sheet-sanitizers** `string[]`: **identificadores** de sanitizers a nivel hoja (ej. `['normalize-headers']`).  
  - **row-validators** `string[]`: **identificadores** de validadores a nivel fila (ej. `['required-fields']`).  
  - **row-sanitizers** `string[]`: **identificadores** de sanitizers a nivel fila (ej. `['trim-all-cells']`).  
  - **row-transformations** `string[]`: **identificadores** de transformaciones a nivel fila (ej. `['calculate-total']`).  
  - **sheet-transformations** `string[]`: **identificadores** de transformaciones a nivel hoja (ej. `['sort-by-date']`).  
  - **fields**: objeto de definición de campos; cada campo tiene:  
    - `name`, `description?`, `example?`  
    - **validators** `string[]`: **identificadores** de validadores de celda (ej. `['required', 'email']`). El engine resuelve desde el Registry y devuelve `SheetError[]` (code + params); el engine los asigna a la celda.  
    - **sanitizers** `string[]`: **identificadores** de sanitizers de celda (ej. `['trim', 'lowercase']`).  
    - **transformations** `string[]`: **identificadores** de transformaciones de celda (ej. `['toUpperCase', 'formatCurrency']`).  
    - **valueType**: `'number' | 'string' | 'bool' | 'date'` — incluye `date` para que el Sanitizer sepa si convertir números de serie de Excel a `Date` o dejarlos como number.  
    - **inputType**: `'input' | 'checkbox'` (u otros según UI)

**Validadores/Sanitizers/Transformadores con parámetros:** Si una función necesita parámetros (ej. rango min-max), el layout puede usar un objeto en lugar de string:

```ts
validators: [
  'required',
  { name: 'range', params: { min: 10, max: 100 } }
]
```

El Registry resuelve `'range'` y el Worker pasa `params` a la función.

**Construcción del layout con Register():** Para facilitar la construcción del layout sin "magic strings", cada controlador (built-in o custom) puede exponer una función **`Register(registerFn?)`** que devuelve el identificador string y, si se pasa el callback de registro del Provider, registra la implementación en el Worker si aún no está registrada. Así el layout puede mezclar:

- **Built-in:** `validators: [CellEmailValidator.Register(registerValidator), 'phone']` — `Register(registerValidator)` devuelve `'email'` y asegura que el validador de email esté registrado; `'phone'` es un id que el usuario debe haber registrado aparte.
- **Custom:** El usuario crea sus propios controladores siguiendo la interfaz expuesta por la librería y usa `MyValidator.Register(registerValidator)` en el layout. Ver *Interfaz para controladores creados por el usuario* más abajo y **Architecture.md** — *Register() pattern*.

Estos tipos se crearán en archivos bajo `src/types/` (por ejemplo `error.ts`, `raw-sheet.ts`, `sheet.ts`, `sheet-layout.ts`, `registry.ts`) y se reexportarán desde un barrel `src/types/index.ts` para uso en parser, provider, validadores y UI. **En este paso solo se actualiza la documentación; la implementación se hará cuando corresponda.**

## Registro agnóstico de validadores/sanitizers/transformadores (Zero-Bundle-Size)

**Objetivo:** Permitir que el usuario **solo importe y registre** las funciones que usa. Si no registra el validador de email, ese código no se incluye en el bundle final (Tree Shaking perfecto). La librería **no registra nada por defecto**; es el usuario quien decide qué "encender".

### Arquitectura del Registry

**1. Registry Core (Universal):** Motor que guarda funciones por nombre y **tipo de nivel** (`cell` | `row` | `table`) para que el Runner en el Worker sepa **cuándo** invocar cada una (por celda/fila o una sola vez al final). No conoce lógica de negocio; solo mapea `string → { fn, type }`.

```ts
// src/core/shared/registry.ts
export type RegistryLevel = 'cell' | 'row' | 'table';

export type RegistryEntry<T = (...args: any[]) => any> = {
  fn: T;
  type: RegistryLevel;  // Runner: cell/row por cada fila; table una vez al final
};

export class Registry<T extends (...args: any[]) => any> {
  private entries = new Map<string, RegistryEntry<T>>();

  register(name: string, fn: T, options: { type: RegistryLevel }): void {
    this.entries.set(name, { fn, type: options.type });
  }

  get(name: string): RegistryEntry<T> | undefined {
    return this.entries.get(name);
  }

  getByType(type: RegistryLevel): Array<{ name: string; entry: RegistryEntry<T> }> {
    return [...this.entries.entries()]
      .filter(([, e]) => e.type === type)
      .map(([name, entry]) => ({ name, entry }));
  }

  has(name: string): boolean {
    return this.entries.has(name);
  }

  clear(): void {
    this.entries.clear();
  }
}
```

El **Register()** de cada controlador (built-in o custom) debe pasar el **type** correcto al registrar (ej. `CellEmailValidator` → `type: 'cell'`, `TableUniqueValidator` → `type: 'table'`). Ver **Architecture.md** — *Registry entry type* y *Table-level validations*.

**2. Building Blocks (Utils):** La librería expone **funciones sueltas** en `src/utils/controller/[context]/`. El usuario elige qué importar.

```ts
// src/utils/controller/email/cell-email-validator.ts
export const emailValidator = (value: any): boolean => 
  /^\S+@\S+\.\S+$/.test(value);

// src/utils/controller/required/cell-required-validator.ts
export const requiredValidator = (value: any): boolean => 
  value !== undefined && value !== null && value !== '';
```

**3. Handshake del Usuario (Integración):** El usuario registra manualmente en su app. Si no llama a `register`, el código de esa función no se incluye en el build. El **Provider** es el cerebro (origen de la verdad); los **Hooks** son la interfaz. Se usan **eventos (EventTarget)** para progreso; **no Zustand**. Ver *Provider as brain, Hooks as interface* en Architecture.md.

```ts
// Configuración: Provider envuelve la sección de importación
import { ImporterProvider, useImporter, useImporterStatus, useSheetData, useSheetEditor } from 'react-import-sheet-headless';
import { emailValidator, requiredValidator } from 'react-import-sheet-headless/utils';

const myLayout = {
  fields: [
    { key: 'user_email', validators: ['required', 'email'] }
  ]
};

function App() {
  return (
    <ImporterProvider>
      <MyUploader />
      <MyDataTable />
    </ImporterProvider>
  );
}

function MyUploader() {
  const { processFile, registerValidator } = useImporter({ layout: myLayout });
  const { status } = useImporterStatus();

  React.useEffect(() => {
    registerValidator('email', emailValidator);
    registerValidator('required', requiredValidator);
  }, [registerValidator]);

  const handleUpload = (file: File) => processFile(file);
  return <input type="file" onChange={(e) => e.target.files?.[0] && handleUpload(e.target.files[0])} />;
}

function MyDataTable() {
  const { sheet, errors } = useSheetData();
  const { editCell } = useSheetEditor();
  return (/* render table with sheet, errors; call editCell on edit */);
}
```

### Serialización para el Worker (Bridge)

Como el usuario registra la función en el **Main Thread**, pero la validación/sanitización/transformación ocurre en el **Worker**, necesitamos un **bridge**:

**Opción A (Recomendada): Código compartido (ESM Imports en Worker):**
- El Worker **importa directamente** el mismo `Registry` y las mismas funciones desde `src/utils/controller/`.
- El Main Thread envía al Worker solo el **nombre** de las funciones a registrar (o el Worker las registra todas al inicio si se decide).
- **Ventaja:** No hay serialización; el código ya está disponible en el Worker. El Worker y el Main Thread comparten el mismo código fuente.
- **Desventaja:** El Worker debe tener acceso al mismo código (no es problema con bundlers modernos que soportan Workers como módulos ESM).

**Opción B: Serialización con `fn.toString()` + `new Function`:**
- Al registrar, el Provider convierte la función a string (`fn.toString()`) y la envía al Worker vía Comlink.
- El Worker usa `new Function` para reconstruir la función en su Registry local.
- **Ventaja:** Funciona incluso si el Worker no tiene acceso directo al código fuente.
- **Desventaja:** Más complejo; `new Function` tiene limitaciones (no puede capturar closures); **solo funciones puras**. Si una función depende de una variable externa (ej. una lista en el scope), esa variable no existe en el Worker y la función reconstruida falla.

**Nota técnica: funciones puras y datos "externos" (listas, umbrales, etc.).** Tanto si se usa Opción A como Opción B, los validadores/sanitizers/transforms que necesiten **datos de configuración** (ej. "el valor debe estar en esta lista") no deben usar variables externas en un closure. **Solución: pasar toda la configuración por `params` en el layout.** La firma del validador recibe `(value, row, params?)`; el layout pasa `{ name: 'in-list', params: { list: ['active', 'paused'] } }`. El validador es puro: todo lo que necesita entra por argumentos. Ejemplo:

```ts
// Función pura. La lista viene por params, no por closure.
const isInListValidator = (value: unknown, row: RawSheetRow, params?: { list: unknown[] }) => {
  if (!params?.list) return null;
  return params.list.includes(value) ? null : [{ code: 'NOT_IN_LIST', params: { value } }];
};
// En el layout:
validators: [{ name: 'in-list', params: { list: ['active', 'paused'] } }]
```

El Worker recibe el layout (con `params`) por structured clone y llama a la función con `(value, row, params)`. Así se puede usar "valor en lista" (y cualquier otra configuración) de forma serializable y compatible con ambas opciones (ESM o fn.toString()). Ver **Architecture.md** — *Technical note: Serialization to Worker and pure functions*.

**Decisión para este proyecto:** Usar **Opción A** (código compartido). Los Workers modernos (con Vite/Webpack 5+) soportan importar módulos ESM. El Worker importa `Registry` y las funciones desde `src/utils/controller/`; el Main Thread solo envía la **lista de nombres** a registrar (o el Worker registra todo lo que el layout necesita al inicio). Si en el futuro se usa Opción B, las funciones deben ser puras y la configuración ir siempre por **params**.

### Ciclo de vida del Registro

**1. Handshake Inicial (una sola vez):**
- El usuario registra funciones al inicio del ciclo de vida de la app (ej. en `useEffect` del Provider o en un componente de setup).
- El Provider (Main Thread) envía al Worker la **lista de nombres** registrados (o el Worker lee el layout y registra solo lo que necesita desde sus imports).

**2. Referencia por Nombre (en el bucle):**
- Cuando el Worker procesa las 10,000 filas, el `sheetLayout` solo lleva strings (`validators: ['required', 'email']`).
- El Worker busca en su Registry local (búsqueda instantánea en un `Map`) y ejecuta la función.

**3. Payload Mínimo:**
- El mensaje que viaja del Main Thread al Worker por cada lote de filas es minúsculo: solo strings cortos, no bloques de código.
- La "reconstrucción" (o import) de la función ocurre **una sola vez** al inicio, no en cada celda.

### Preset Helpers (Opcional pero recomendado)

Para que el usuario no tenga que registrar uno por uno, se puede crear una función "combo":

```ts
// src/utils/presets/standard-validators.ts
import { emailValidator } from '../controller/email/cell-email-validator';
import { requiredValidator } from '../controller/required/cell-required-validator';
// ... otros

export const registerStandardValidators = (register: (name: string, fn: any) => void) => {
  register('email', emailValidator);
  register('required', requiredValidator);
  // ... otros
};
```

El usuario puede llamar `registerStandardValidators(registerValidator)` para registrar todos de un golpe. El Tree Shaking sigue funcionando porque esta función vive en `utils` y solo se importa si el usuario la usa.

### Ubicación en el proyecto

- **Registry Core:** `src/core/shared/registry/` (o `src/core/shared/registry.ts`). Usado por Validator, Sanitizer y Transform.
- **Building Blocks:** `src/utils/controller/[context]/` (ej. `email/`, `required/`, `phone/`, `date/`). Cada contexto exporta funciones sueltas.
- **Preset Helpers:** `src/utils/presets/` (ej. `standard-validators.ts`, `standard-sanitizers.ts`, `standard-transforms.ts`).
- **API en Provider/Hooks:** **`useImporter({ layout })`** expone **`processFile(file)`**, `registerValidator`, `registerSanitizer`, `registerTransform`, `abort`. **`useImporterStatus()`** expone status y progress. **`useSheetData()`** expone sheet y errors. **`useSheetEditor()`** expone `editCell`. Los register reciben `options: { type: 'cell' | 'row' | 'table' }`. Internamente el Provider mantiene tres Registries (`{ fn, type }`) y los sincroniza con el Worker.

### Ventajas de este diseño

- **Zero-Bundle-Size inicial:** Si el usuario no usa el validador de email, ese código nunca llega al navegador del cliente final.
- **Modularidad total:** El sistema es totalmente modular; el usuario puede crear sus propios validadores/sanitizers/transformadores y registrarlos de la misma forma.
- **Seguridad:** Al estar ya registrada, se evita el uso de `eval` o `new Function` dentro del bucle de alta frecuencia. La "reconstrucción" (o import) ocurre una sola vez al inicio.
- **Consistencia:** Si el usuario usa la misma validación en 5 columnas diferentes, el Worker solo guarda una copia de la función en memoria y la reutiliza.
- **Madurez 10/10:** Separación clara entre "definición de la lógica" (Registro) y "aplicación de la lógica" (Ejecución).

### Patrón Register() (built-in y custom)

Cada controlador (validador, sanitizer, transform — celda, fila o hoja) **expone una función `Register(registerFn?)`** que:

1. **Devuelve el identificador string** de ese controlador (ej. `'email'`, `'required'`). Es el mismo string que se usa en el layout y en el Registry del Worker.
2. **Opcionalmente registra la implementación** en el Provider/Worker cuando se pasa el callback. Si el usuario llama `CellEmailValidator.Register(registerValidator)`, la librería registra el validador de email (y sincroniza con el Worker si aún no está registrado) y devuelve `'email'`. El layout puede mezclar: `validators: [CellEmailValidator.Register(registerValidator), 'phone', MyCustomValidator.Register(registerValidator)]`.
3. **Registro idempotente:** Si el controlador ya estaba registrado (p. ej. por un preset), `Register(registerFn)` no vuelve a registrar; solo devuelve el identificador.

**Controladores built-in** (en `src/utils/controller/[context]/`) exportan la función cruda y el helper Register, ej.:

- `CellEmailValidator.validate` — función a ejecutar.
- `CellEmailValidator.id` — constante string `'email'`.
- `CellEmailValidator.type` — constante **`'cell'`** (para que el Registry y el Runner sepan que se invoca por celda).
- `CellEmailValidator.Register(registerValidator?: (name: string, fn: T, options?: { type: RegistryLevel }) => void): string` — devuelve `'email'`; si se pasa `registerValidator`, registra `CellEmailValidator.validate` bajo `'email'` **con `type: 'cell'`** (si no estaba registrado) y devuelve `'email'`. Análogamente, los controladores de fila/hoja registran con `type: 'row'` o `type: 'table'`.

**Interfaz para controladores creados por el usuario:** La librería expone **interfaces** (y opcionalmente helpers de factory) para que el usuario defina sus propios validadores/sanitizers/transforms con el mismo contrato `Register()`. Ejemplo conceptual para validadores de celda:

```ts
// src/types/controller-contracts.ts (o similar)
type RegistryLevel = 'cell' | 'row' | 'table';

interface CellValidatorDefinition {
  readonly id: string;
  readonly type: RegistryLevel;  // 'cell' para este caso
  validate: CellValidatorFn;      // (value, row) => SheetError[] | null
  Register(registerFn?: (name: string, fn: CellValidatorFn, options: { type: RegistryLevel }) => void): string;
}

// El usuario puede crear (y debe pasar type al registrar para que el Runner sepa cuándo invocar):
const MyCustomValidator: CellValidatorDefinition = {
  id: 'my-rule',
  type: 'cell',
  validate(value, row) { /* ... */ return errorsOrNull; },
  Register(registerFn) {
    if (registerFn) registerFn(this.id, this.validate, { type: this.type });
    return this.id;
  },
};
```

Así el usuario usa en el layout `validators: [CellEmailValidator.Register(registerValidator), MyCustomValidator.Register(registerValidator)]`. Se expondrán interfaces análogas para sanitizers y transforms (cell, row, sheet) en los pasos 5, 6 y 7.

**Sync vs async en los tipos:** Las funciones de **cell** y **row** (validators, sanitizers, transforms) deben ser **síncronas** (retorno directo, no `Promise`). Solo las de **table** pueden ser **async** (retorno `Promise<SheetError[] | null>` para validators, o `Promise<...>` para transforms con resultado actualizado). Así el Runner sabe que en el bucle solo invoca sync y al final puede hacer `await` solo para los table. Ver **Architecture.md** — *Sync vs async* y **6. Validator** / **7. Transform**.

## Plan de pasos

1. **Definir el estado del importer (solo estado estable)**
   - Tipo `ImporterState` con:
     - `file: File | null` — archivo seleccionado (o ninguno).
     - `rawData: RawSheet | null` — datos crudos parseados; **nunca** `any[]`. Si el parser no ha terminado es `null`; si terminó, debe cumplir estrictamente la interfaz `RawSheet`. Así se mantienen las ventajas de TypeScript en el corazón del Provider.
     - `documentHash: string | null` — identificador único generado por el Parser (p. ej. hash del contenido o del archivo). Propiedad de primer nivel para que los hooks de persistencia (IndexedDB, etc.) puedan reaccionar en cuanto cambie el archivo.
     - `status` — estado de la operación (p. ej. `'idle' | 'loading' | 'success' | 'error'`).
     - `result` (sheet) y errores cuando corresponda (según pasos posteriores).
   - **No** incluir `progress` en el estado: el progreso cambia cada milisegundo y provocaría re-renders masivos en todos los consumidores del Context. Ver *Progreso vía EventTarget* más abajo.
   - Tipo `ImporterStatus` para los valores de `status`.

2. **Crear el contexto y el provider (ImporterProvider = cerebro)**
   - **Provider como Single Source of Truth:** **ImporterProvider** envuelve la sección de la app donde se realiza la importación. Gestiona el ciclo de vida de los Workers y el estado global. **No se usa Zustand**; estado en React state y progreso vía **EventTarget** (eventos).
   - **Layout:** El usuario pasa el layout al hook **`useImporter({ layout })`**; el hook lo inyecta en el Provider (p. ej. `setLayout` en efecto). El Provider almacena el layout y lo expone en el contexto. Opcional: el Provider puede aceptar un prop **`layout`** inicial.
   - **Estado y archivo:** El Provider mantiene `ImporterState` (file, rawData, documentHash, status, result). Si el usuario navega entre pestañas (Carga → Mapeo → Validación), el archivo y el progreso no se pierden; no hay prop drilling — cualquier hijo usa los hooks.
   - **Workers:** El Provider inicializa los Workers **una sola vez** (p. ej. en `useEffect`), mantiene las instancias de Comlink y orquesta Parse → Convert → Sanitize → Validate → Transform. Puede hacer **gestión de estados finitos** (p. ej. no validar si el parseo no ha terminado). Al desmontar, hace cleanup (termina Workers).
   - **EventTarget:** Un **EventTarget** dispara **`importer-progress`** y **`importer-aborted`**. Solo la UI de progreso se suscribe. **Registries** (tres); **`abort()`**; en abort o unmount, **`worker.terminate()`**, reset status, **`importer-aborted`**.

3. **Hooks como interfaz (cuatro hooks especializados)**
   - **`useImporter({ layout })`** — Punto de entrada. Recibe el **layout** y lo pasa al Provider. Expone **`processFile(file)`** (dispara el pipeline vía Provider), **`registerValidator`**, **`registerSanitizer`**, **`registerTransform`**, **`abort`**. Error claro si se usa fuera del Provider.
   - **`useImporterStatus()`** — Devuelve **status** (idle, parsing, validating, success, error) y forma de leer **progress** (suscripción al EventTarget o snapshot). Para la barra de progreso.
   - **`useSheetData()`** — Devuelve el **resultado** (sheet final) y los **errores** para renderizar la tabla. El componente no conoce Workers.
   - **`useSheetEditor()`** — Expone **`editCell`** para el pipeline de edición.
   - Opcional: **`useImporterEventTarget()`** — EventTarget o **`subscribeToProgress(callback) => unsubscribe`** con cleanup para evitar memory leaks.

4. **Tests unitarios (Vitest)**
   - Tests para:
     - Estado inicial del provider (`file: null`, `rawData: null`, `documentHash: null`, `status: 'idle'` o equivalente).
     - Que `useImporter` (y el resto de hooks) lanzan (o devuelven error) cuando se usan fuera del ImporterProvider.
     - Que los hijos del provider reciben el estado y pueden actualizarlo (setFile, setRawData, setStatus) si se exponen.
     - Que el EventTarget expuesto dispara eventos con el detail esperado cuando se invoca (mock del callback de progreso).
     - **Registry:** Que `registerValidator` / `registerSanitizer` / `registerTransform` agregan funciones al Registry correspondiente; que `get(name)` devuelve la función registrada; que si se intenta usar un identificador no registrado, el sistema emite error claro (ej. `VALIDATOR_NOT_FOUND`).
     - **Abort y cleanup:** Que al llamar `abort()` el Worker activo se termina y el status pasa a `idle` (o `cancelled`); que se despacha `importer-aborted`; que al desmontar el Provider (o el hook que tiene el Worker) se ejecuta el cleanup y se llama `terminate()` sobre el Worker.

## Progreso vía EventTarget (evitar re-renders masivos)

El **progreso** (p. ej. fila 350 de 10.000) no se guarda en el Context porque cambiaría constantemente y re-renderizaría todos los componentes que consumen el contexto (tablas, formularios, botones). En su lugar:

- El Worker (o la capa que recibe el callback de progreso del Worker) **dispara un evento personalizado** en un **EventTarget** (p. ej. `importer-progress` con `detail: { phase, globalPercent, localPercent, currentRow, totalRows }`).
- Solo el componente que muestra la barra de progreso se suscribe a ese evento y actualiza su ancho/vista mediante **ref** (manipulación imperativa del DOM) o un **estado local** muy pequeño (`useState` solo en ese componente). El resto del árbol React no se entera del avance.
- **Ciclo de vida**: si el EventTarget vive en un ref del Provider o es una instancia global, los componentes que se suscriban a `importer-progress` **deben desuscribirse al desmontarse**. El hook `useImporterEventTarget` debe ofrecer una API de suscripción con cleanup (p. ej. devolver una función `subscribe(callback) => unsubscribe`) o documentar de forma explícita que el usuario debe eliminar el listener en el cleanup de `useEffect` para evitar memory leaks.
- No se usa Zustand ni otras librerías externas; solo `EventTarget` y `CustomEvent` del navegador. Ver **Architecture.md** — *Progress and re-renders: EventTarget*.

## Eficiencia de datos (estrategia Delta)

Para no duplicar memoria ni bloquear el hilo principal con clones grandes, el Worker recibe el archivo/RawSheet **una sola vez**, ejecuta Sanitize → Validate → Transform internamente, y en lugar de devolver la sheet completa en cada paso devuelve **errores** (lista compacta) y opcionalmente **deltas** (solo filas/celdas cambiadas). El main thread mantiene una copia local de la sheet y la **parchea** al recibir errores/deltas. Ver **Architecture.md** — *Data efficiency: Delta strategy* y *Maturity & resilience → Memory management for massive files*.

## Resiliencia (timeout y recuperación)

El Provider y los hooks que invocan Workers deben contemplar **timeout** y **recuperación**: si un Worker no responde en X segundos (configurable), emitir error controlado (ej. `code: 'WORKER_TIMEOUT'`) y permitir **reintentar** sin recargar la página. Ver **Architecture.md** — *Maturity & resilience → Worker timeout & recovery*.

## Abort y ciclo de vida (Worker cleanup)

Si el usuario **cancela** la importación o **desmonta** el componente (p. ej. cierra el modal o cambia de página) mientras un Worker está procesando, el Worker debe **dejar de ejecutarse** y liberar recursos. De lo contrario hay fuga de CPU/memoria y en móvil se drena la batería. Ver **Architecture.md** — *Maturity & resilience → Process Abort & Worker Cleanup*.

**Requisitos:**

- **API `abort()`:** El Provider (o el hook que orquesta el pipeline) expone **`abort()`** (o **`cancel()`**). Al llamarla: (1) se invoca **`worker.terminate()`** sobre el Worker activo; (2) se actualiza **status** a `'idle'` (o `'cancelled'` si se define); (3) se despacha el evento **`importer-aborted`** en el EventTarget para que la barra de progreso deje de actualizarse; (4) se limpia la referencia al Worker para que la siguiente importación cree uno nuevo.
- **Cleanup al desmontar:** En el Provider (o en cada hook que instancie un Worker, p. ej. `useParserWorker`, `useValidatorWorker`), el **return del `useEffect`** debe terminar el Worker si existe: `return () => { if (workerRef.current) workerRef.current.terminate(); workerRef.current = null; };`. Así, si el usuario usa el importer en una página y navega a otra, el Worker se mata al desmontar.
- **Evento `importer-aborted`:** Contrato del evento (nombre y opcionalmente `detail`) documentado para que la UI pueda escucharlo y ocultar o resetear la barra de progreso.

Los pasos 3 (Parser), 5 (Sanitizer), 6 (Validator) y 7 (Transform) deben almacenar la instancia del Worker en una ref accesible por el Provider (o por un worker manager) para que `abort()` y el cleanup del Provider puedan llamar `terminate()`.
