# 11. Telemetry & World-Class Features — Plan de construcción

## Objetivo

Implementar **telemetría de procesamiento** (métricas por fase del pipeline) y tres características "clase mundial" que mejoran la experiencia del consumidor: **fuzzy header mapping**, **virtualización de datos** (API de paginación) y **persistencia de estado** con IndexedDB. Todo se integra en la arquitectura headless existente (Provider, hooks, Workers). El plan eleva el paquete de "utilidad de importación" a **infraestructura de datos profesional**, cubriendo: **Visibilidad** (métricas), **Resiliencia/UX** (fuzzy + persistencia) y **Escalabilidad** (virtualización).

---

## Contexto en el flujo

- **2. Setting.md** define `ImporterState`, Provider y `useImporter()`.
- **3. Parser** → **4. Convert** → **5. Sanitizer** → **6. Validator** → **7. Transform** se ejecutan (Convert en main thread; el resto en Web Workers) (cada uno o un pipeline unificado según implementación).
- **8. Edit** ya define `getPaginatedResult(page, pageSize)`; este paso (11. Telemetry) unifica y amplía la API de acceso a filas para virtualización.
- La **telemetría** no es solo visualización: es la **herramienta de depuración** del desarrollador (dashboard de rendimiento). Se mide con `performance.now()` cada fase dentro del/los Worker(s), más el overhead de serialización en el Main Thread, y se expone un objeto de métricas en el Provider.

# Parte A — Telemetría de procesamiento (métricas)

## Objetivo A

Medir el rendimiento de **cada paso** que se ejecuta en el/los Worker(s): **parse**, **sanitize**, **validate**, **transform**. Agregar los tiempos y exponerlos en el Provider para que el consumidor pueda mostrar estadísticas (ej. "Importación completada en 1,2 s — 0,05 ms/fila").

## Tipos de métricas

Definir en **`src/types/metrics.ts`** (o en `src/types/index.ts`):

```ts
/** Tiempos en ms por fase del pipeline (raw numbers from performance.now()). */
export interface PipelineMetricsTimings {
  parse: number;
  sanitize: number;
  validate: number;
  transform: number;
}

/** Desglose porcentual por fase (0–100). Útil para ver el cuello de botella de un vistazo. */
export interface PipelineMetricsPercentages {
  parse: number;
  sanitize: number;
  validate: number;
  transform: number;
  /** Tiempo fuera del Worker (serialización + mensajes). */
  overhead?: number;
}

/** Métricas expuestas al consumidor (strings para visualización + números para cálculos). */
export interface PipelineMetrics {
  /** Tiempo por fase en ms (número). */
  timings: PipelineMetricsTimings;
  /** Desglose porcentual: ej. "80% en validaciones" → validaciones son el cuello de botella. */
  percentages?: PipelineMetricsPercentages;
  /** Tiempo total en ms. */
  totalMs: number;
  /** true si totalMs > umbral (ej. 2000). Permite al desarrollador saber al instante si la importación fue lenta. */
  isSlow: boolean;
  /** Para UI: "1.23ms", "45.67ms", etc. */
  parseTime: string;
  sanitizeTime: string;
  validateTime: string;
  transformTime: string;
  totalTime: string;
  /** Eficiencia: ms por fila (totalMs / rowCount). */
  efficiency: string;  // e.g. "0.0523ms/row"
  /** Número de filas usadas para efficiency (rowsCount del RawSheet o sheet.rows.length). */
  rowCount: number;
  /** Tiempo en ms gastado en overhead (Main Thread: envío/recepción de mensajes, serialización). Opcional. */
  overheadMs?: number;
}
```

- Los Workers devuelven **números** (ms); el orquestador (main thread o worker unificado) construye `PipelineMetrics` y lo guarda en el estado.
- Si un paso no se ejecuta (ej. sin sanitizers), su tiempo será `0`.
- **isSlow:** calcular como `totalMs > SLOW_THRESHOLD_MS` (ej. 2000). Configurable o constante.
- **percentages:** si `totalMs > 0`, cada fase = `(timings.phase / totalMs) * 100`. Incluir `overhead` si se mide.

## Dónde medir

- **Dentro de cada Worker:** `t0 = performance.now()` antes de la fase; al salir `delta = performance.now() - t0`. Incluir en el mensaje de retorno (ej. `{ rawSheet, metrics: { parse: delta } }`).
  - Parser: antes/después de `parser.parse(file)`.
  - Sanitizer / Validator / Transform: igual alrededor de cada paso.
- Si el diseño usa **un solo Worker** que ejecuta todo el pipeline internamente:
  - Dentro de ese Worker: `const tParse = performance.now(); ... parse(...); metrics.parse = performance.now() - tParse;` (idem sanitize, validate, transform).
  - Devolver `{ data: sheet, metrics }` con `timings` y, opcionalmente, strings formateados.

**Overhead de serialización (Main Thread):** para que las métricas reflejen el coste real que ve el usuario (incluyendo paso de mensajes entre hilos), el **main thread** debe:

1. `tStart = performance.now()` justo **antes** de enviar el mensaje al Worker (postMessage o equivalente).
2. Al recibir la respuesta (onmessage), `tEnd = performance.now()`; `wallClockMs = tEnd - tStart`.
3. Los tiempos internos del Worker suman `workerTotalMs = parse + sanitize + validate + transform`.
4. `overheadMs = wallClockMs - workerTotalMs` (puede ser negativo por precisión; acotar a `>= 0` si se desea). Incluir en `PipelineMetrics.overheadMs` y en `percentages.overhead` para visibilidad (ej. "5% overhead").

El **main thread** debe además:

1. Recibir o agregar los tiempos de cada paso.
2. Calcular `totalMs` (usar `wallClockMs` como tiempo total percibido, o `workerTotalMs` si se prefiere solo tiempo de cómputo), `totalTime` (string), `efficiency` (totalMs / rowCount), `isSlow`, `percentages`.
3. Dispatch al Provider: `SET_METRICS` con `PipelineMetrics`.
4. El Provider guarda `state.metrics` y lo expone vía `useImporter().metrics`.

## Estado en el Provider

- En **`ImporterState`** añadir:
  - `metrics: PipelineMetrics | null` — métricas de la última ejecución completa del pipeline (null hasta que termine con éxito una importación).
- En **`useImporter()`** devolver:
  - `metrics: state.metrics` (solo lectura).

## Criterios de éxito (Telemetría)

- Cada fase (parse, sanitize, validate, transform) tiene su tiempo medido con `performance.now()` dentro del Worker.
- Opcional: overhead de Main Thread (wallClock − workerTotal) calculado y expuesto como `overheadMs` / `percentages.overhead`.
- El resultado incluye `PipelineMetrics` con `timings`, `totalMs`, `isSlow`, `percentages` (desglose %), `totalTime`, `efficiency`, `rowCount`.
- El Provider almacena y expone `metrics` en el estado y en el hook.
- Tests: mock de workers que devuelven tiempos fijos; comprobar forma de `metrics`, cálculo de `isSlow` y `percentages`.

# Parte B — Fuzzy Header Mapping

## Objetivo B

Permitir emparejar headers del archivo (raw) con las keys esperadas del layout aunque no coincidan exactamente (espacios, mayúsculas, tildes, typos). Usar **distancia de Levenshtein** (o similar) y un umbral (ej. score > 0.8). Esta "magia" de UX mejora la resiliencia y ahorra tiempo al usuario en las demos.

## Normalización previa

Antes del fuzzy match, **siempre** normalizar para reducir falsos negativos sin cálculos pesados:

- `.trim().toLowerCase()` en ambas cadenas.
- Eliminar tildes/acentos (normalizar a NFD y quitar rangos de combinación, o tabla de reemplazo). Ej.: "Nombre" → "nombre", "Teléfono" → "telefono".

Esto resuelve la mayoría de los casos (espacios, mayúsculas, acentos) antes de aplicar Levenshtein.

## Implementación

1. **Utilidad de similitud** — **`src/core/parser/utils/fuzzyMatch.ts`**

   - Función auxiliar `normalize(s: string): string`: trim, toLowerCase, quitar acentos.
   - Implementar `getSimilarity(s1: string, s2: string): number` (Levenshtein normalizado; devuelve 0–1; 1 = idéntico). Comparar sobre strings ya normalizados.

   Estructura recomendada para `getSimilarity`:

   ```ts
   export const getSimilarity = (s1: string, s2: string): number => {
     const longer = s1.length < s2.length ? s2 : s1;
     const shorter = s1.length < s2.length ? s1 : s2;

     if (longer.length === 0) return 1.0;

     const distance = levenshteinDistance(longer, shorter);
     return (longer.length - distance) / longer.length;
   };

   // Implementación de Levenshtein (matriz o versión optimizada en espacio);
   // puede ser dependencia ligera: fastest-levenshtein, o propia.
   ```

   Opción: dependencia ligera (`fastest-levenshtein`) o implementación propia (matriz de distancia).

2. **Mapeo de headers — Estrategia "Gula con prioridad"**

   Para evitar que dos columnas mapeen a la misma key (ej. "Nombre" y "Nombre Completo" ambos → `name`):

   - Calcular **todos** los pares (expectedKey, rawHeader) con su score: `getSimilarity(normalize(key), normalize(header))`.
   - Ordenar la lista de pares por **score descendente**.
   - Asignar el par con **score más alto** (y solo si score ≥ THRESHOLD, ej. 0.8): ese key recibe ese header.
   - Eliminar ese **key** y ese **header** de los candidatos y **repetir** hasta agotar pares por encima del umbral o candidatos.

   Función `mapHeaders(rawHeaders: string[], expectedKeys: string[], options?: { threshold?: number }): { key: string; matchedHeader: string | null }[]`:

   - Resultado: un elemento por cada `expectedKey`, con `matchedHeader` asignado o `null` si no hubo match por encima del umbral.

3. **Integración**

   - Usar después del **parsing** (Worker del parser o main thread con RawSheet):
     - `rawHeaders = rawSheet.headers`; `expectedKeys = sheetLayout.fields` (o array de keys).
     - Llamar `mapHeaders(rawHeaders, expectedKeys)` y usar el resultado para construir header → key (o reasignar `cell.key` en cada fila).
   - Documentar que el consumidor puede usar este mapeo para sugerir o aplicar automáticamente columnas → campos.

## Criterios de éxito (Fuzzy)

- Normalización (trim, lowerCase, sin acentos) aplicada antes de comparar.
- `getSimilarity` devuelve 1 para strings idénticos (tras normalizar) y valores menores para cadenas distintas.
- `mapHeaders` usa asignación "gula con prioridad": cada key y cada header se asigna como máximo una vez; el mejor score primero.
- Tests unitarios para `getSimilarity` y `mapHeaders` (umbral, emparejamiento único, casos con acentos y espacios).

# Parte C — Virtualización de datos (paginación / getRows)

## Objetivo C

Exponer una API de **alto rendimiento** que permita al consumidor no depender del array completo en memoria para renderizar (p. ej. con `react-window` o `@tanstack/react-virtual`). El headless ofrece **total de filas** y **getRows(page, limit)** como primitivos para virtualización.

## Implementación

- En **`useImporter()`** (o en el hook que expone el resultado del pipeline):
  - Exponer:
    - `totalRows: number` — derivado de `sheet.rows.length` (0 si no hay sheet). Debe reflejar el **conjunto actualmente visible** si en el futuro se implementan filtros (reactive pagination: si el usuario filtra la vista, `totalRows` debe actualizarse para que el scroll virtual y el número de páginas sean correctos).
    - `getRows(page: number, limit: number): Row[]` — devolver el slice sobre el mismo conjunto que usa `totalRows` (p. ej. `sheet.rows.slice((page - 1) * limit, (page - 1) * limit + limit)`; si hay filtros, slice sobre las filas filtradas).
  - Convención: `page` 1-based (página 1 = primeras `limit` filas).
- Opcional: mantener **8. Edit** `getPaginatedResult(page, pageSize)` como wrapper: `{ page, pageSize, totalCount: totalRows, rows: getRows(page, pageSize) }`.

## Contrato y ejemplo para README

```ts
// Desde useImporter()
const { getRows, totalRows, sheet, metrics, ... } = useImporter();

// Ejemplo: virtualización con página 2 y 50 filas
const rows = getRows(2, 50);  // filas 51–100
// totalRows para calcular número de páginas o tamaño del scroll virtual
```

**Documentación (README):** incluir un ejemplo breve de uso de `getRows` + `totalRows` con **react-window** (o `@tanstack/react-virtual`) para dar valor inmediato al usuario, p. ej.:

```tsx
import { FixedSizeList as List } from 'react-window';
const { getRows, totalRows } = useImporter();
const pageSize = 50;
const Row = ({ index, style }) => {
  const page = Math.floor(index / pageSize) + 1;
  const offset = index % pageSize;
  const chunk = getRows(page, pageSize);
  const row = chunk[offset];
  return <div style={style}>{row ? /* render row */ : null}</div>;
};
<List height={400} itemCount={totalRows} itemSize={32} width="100%">{Row}</List>
```

(Ajustar según API real de react-window: itemCount = totalRows, obtener fila por índice vía getRows desde el índice lineal.)

## Criterios de éxito (Virtualización)

- `totalRows` refleja el número de filas del resultado actual (y, si se añaden filtros en el futuro, del conjunto filtrado).
- `getRows(page, limit)` devuelve el slice correcto (vacío si page/limit fuera de rango).
- README incluye ejemplo de integración con react-window (o similar).

# Parte D — Persistencia con IndexedDB ("Inmortalidad de sesión")

## Objetivo D

Permitir guardar y restaurar el estado del importer en **IndexedDB** para evitar el límite de **5MB de LocalStorage** (insuficiente para CSVs grandes, ej. 50.000 filas) y permitir sesiones grandes o recuperación tras cierre del navegador. IndexedDB es el estándar adecuado para datos voluminosos en el navegador.

## Estado serializable: no guardar `File`

**Importante:** el objeto **`File`** no es fiable para guardar directamente en IndexedDB (no todos los navegadores lo serializan bien). Es más seguro:

- **No** guardar el `File` tal cual; o
- Guardar **ArrayBuffer** (contenido del archivo) o el **string** ya procesado (raw CSV/texto), o
- Guardar solo el **resultado final** (sheet + metadata), que es suficiente para reanudar la sesión sin volver a leer el archivo.

Definir **`SerializableImporterState`** como subconjunto de `ImporterState` sin `File`: p. ej. sheet, status, metrics, y si se necesita "archivo", usar ArrayBuffer/string o omitir.

## Versionado de esquema (Schema versioning)

Si el **sheetLayout** o los identificadores de validadores/transformadores cambian (ej. renombrar `email` → `is-email`), las sesiones ya persistidas en IndexedDB pueden volverse incompatibles. Ver **Architecture.md** — *Maturity & resilience → Schema versioning*.

**Requisitos:**

- **Versión en el estado persistido:** Al guardar una sesión, incluir siempre la **versión del layout** (p. ej. `sheetLayout.version`) y, si se define, una **versión del esquema del estado** (ej. `stateSchemaVersion: 1`) en el objeto que se escribe en IndexedDB. Así al cargar se puede comparar con el layout actual.
- **Al cargar:** Si `loadSession(id)` devuelve datos con una versión de layout distinta a la del `sheetLayout` actual del Provider, el sistema debe:
  - **Opción A:** No restaurar automáticamente y notificar al consumidor (ej. callback `onVersionMismatch?`) para que decida (limpiar sesión, re-importar, o aplicar migración).
  - **Opción B:** Ofrecer una API de **migración** (ej. `migrateSession(oldState, fromVersion, toVersion)`) que el consumidor pueda usar para adaptar datos viejos al nuevo esquema antes de restaurar.
- Documentar en README que al cambiar `sheetLayout.version` o los identificadores de validadores/transformadores, las sesiones guardadas con la versión anterior pueden requerir migración o limpieza (`clearSession`).

## Seguridad y privacidad

Los datos en IndexedDB **persisten** en el dispositivo. Incluir y documentar:

- **`clearSession(id: string): Promise<void>`** — borrar la sesión por id (eliminar del store).
- En la **documentación (README)**, recomendar que el desarrollador llame a `clearSession(persistId)` una vez que la importación se haya **enviado al servidor con éxito**, para no dejar datos sensibles almacenados indefinidamente.

## Implementación

1. **Dependencia**
   - Instalar **`idb`** (wrapper Promise para IndexedDB): `npm install idb`.

2. **Módulo de persistencia** — **`src/core/persistence/`** (o `src/utils/persistence/`)
   - `openDb()`: abrir DB con nombre constante (ej. `'importer-db'`), versión 1, object store `'sessions'`.
   - `saveSession(id: string, state: SerializableImporterState): Promise<void>` — guardar por `id`; el estado debe incluir **layoutVersion** (y opcionalmente **stateSchemaVersion**) para versionado.
   - `loadSession(id: string): Promise<SerializableImporterState | undefined>` — cargar por `id`; al restaurar, comprobar compatibilidad de versión con el layout actual (ver *Versionado de esquema*).
   - **`clearSession(id: string): Promise<void>`** — eliminar sesión por id (recomendado tras envío exitoso al servidor).

3. **Integración en el Provider**
   - Config: `persist: boolean`, `persistId: string`.
   - Tras **éxito** del pipeline: si `persist`, llamar `saveSession(persistId, serializableState)`.
   - En **mount** (useEffect): si `persist` y `persistId`, llamar `loadSession(persistId)`; si hay datos, dispatch `RESTORE_SESSION`. Tener en cuenta efectos secundarios y estado asíncrono en el mount (evitar condiciones de carrera con otras inicializaciones).
   - Decidir si al restaurar se usa `status: 'idle'` o `'success'` y si se expone algo del "archivo" (nombre/metadata) para coherencia de UI.

## Criterios de éxito (Persistencia)

- Guardar sesión tras importación exitosa cuando `persist` está activo; estado guardado es serializable (sin `File` crudo; ArrayBuffer/string/sheet según diseño).
- **Versionado:** Cada sesión guardada incluye `layoutVersion` (de `sheetLayout.version`) y opcionalmente `stateSchemaVersion`; al cargar, se comprueba compatibilidad con el layout actual y se documenta o implementa manejo de desajuste (onVersionMismatch, migración o clearSession).
- Al montar con `persist` + `persistId`, cargar sesión y restaurar estado si existe y la versión es compatible.
- **clearSession(id)** implementado y documentado; README recomienda limpiar tras envío exitoso al servidor.
- Tests: mock de `idb` para put/get/delete con el id correcto; tests de versionado (guardar con version, cargar con layout de otra versión).

# Orden de implementación sugerido (prioridades)

1. **Parte A — Telemetría (Prioridad 1)**  
   La más fácil de implementar y da feedback inmediato sobre el trabajo de los pasos 1–6. Tipos `PipelineMetrics` (con `isSlow`, `percentages`, opcional `overheadMs`), instrumentar Workers y Main Thread, exponer `metrics` en Provider y hook. No depende de B/C/D.

2. **Parte C — Virtualización (Prioridad 2)**  
   Crítica para que el usuario pueda ver sus datos si el archivo es grande. Ampliar `useImporter` con `getRows` y `totalRows`; documentar ejemplo con react-window. Depende de que el Provider exponga `sheet`.

3. **Parte B — Fuzzy header mapping (Prioridad 3)**  
   "Magia" de UX que impresiona en demos. Utilidad con normalización + Levenshtein + asignación "gula con prioridad"; integración en parser/preview. Independiente.

4. **Parte D — Persistencia (Prioridad 4)**  
   La más compleja por manejo de efectos secundarios y estado asíncrono en el mount. Módulo idb (`saveSession`, `loadSession`, **clearSession**), opciones en Provider, restore en mount. Opcional persistir también `metrics`.
