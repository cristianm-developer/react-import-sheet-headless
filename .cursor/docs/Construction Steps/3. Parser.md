# 3. Parser — Plan de construcción

## Objetivo

Construir la primera parte del proyecto: un **parser universal** que reciba archivos de hoja de cálculo (xlsx, xls, ods, csv), los procese en un Web Worker y devuelva siempre el mismo formato JSON normalizado para el resto del flujo.

---

## Alcance

| Entrada | Librería | Extensiones / MIME |
|--------|----------|--------------------|
| Excel / OpenDocument | **SheetJS (xlsx)** | `.xlsx`, `.xls`, `.ods` y MIME correspondientes |
| CSV | **Papa Parse** | `.csv`, `text/csv` |

El parser se ejecuta en un **Web Worker**. El usuario sube el archivo → se envía como **Blob** al worker → se procesa → se devuelven datos en el formato unificado.

**Resiliencia:** Los Workers pueden fallar por memoria, cierre inesperado o bucles infinitos. La capa que invoca al Worker (main thread) debe implementar **timeout y recuperación**: timeout configurable por invocación, error controlado (ej. `WORKER_TIMEOUT` / `WORKER_CRASHED`) y API para reintentar sin recargar la página. Ver **Architecture.md** — *Maturity & resilience → Worker timeout & recovery*.

**Abort y ciclo de vida:** El hook que instancia el Worker del parser (p. ej. `useParserWorker` o `useImportSheet`) debe guardar la instancia en una **ref** accesible para el Provider (o para el `abort()` global), y en el **cleanup de `useEffect`** al desmontar debe llamar **`worker.terminate()`** para evitar fugas de recursos si el usuario cierra el modal o cambia de página. El Provider expone **`abort()`** que termina este Worker y despacha **`importer-aborted`**. Ver **Architecture.md** — *Process Abort & Worker Cleanup* y **2. Setting** — *Abort y ciclo de vida*.

## Formato JSON de salida (contrato) — tipos `raw*`

**Convención de nombres**: todos los tipos de este paso usan el prefijo **`raw*`** (RawSheet, RawSheetRow, RawSheetCell) para mantener la distinción con los tipos que se usarán en los siguientes archivos/pasos del proyecto (datos mapeados o transformados).

Todas las librerías deben normalizarse a esta estructura. Son datos **raw** (sin transformar) del archivo:

```ts
/** Tipos primitivos que SheetJS y Papa Parse ya identifican; facilita al Sanitizer/Validator. */
type RawSheetCellValue = string | number | boolean | Date | null;

interface RawSheetCell {
  key: string;   // clave del header (columna)
  value: RawSheetCellValue; // valor de la celda (primitivos; no unknown)
}

interface RawSheetRow {
  index: number;
  cells: RawSheetCell[];
}

interface RawSheet {
  name: string;
  filesize: number;
  documentHash: string;   // hash del documento para identificación única (ej. SHA-256 en hex)
  rowsCount: number;
  headersCount: number;
  headers: string[];
  rows: RawSheetRow[];
}

/** Salida del Worker: mapa de hojas por nombre. v1 usa la primera hoja; el contrato prevee múltiples desde día 1. */
interface RawParseResult {
  sheets: Record<string, RawSheet>;
}
```

- **name**: nombre de la hoja (o del archivo si es CSV / una sola hoja).
- **filesize**: tamaño del archivo en bytes.
- **documentHash**: **hash del documento** para identificación única (p. ej. SHA-256 del contenido del Blob/archivo, en hex). Permite detectar si es el mismo archivo en re-importaciones o en sesiones distintas. Debe calcularse por **streaming/chunks** para no duplicar memoria en archivos grandes (ver sección *documentHash por streaming*).
- **rowsCount**: número total de filas de datos (excluyendo cabecera si aplica).
- **headersCount**: número de columnas/cabeceras.
- **headers**: array de nombres de columnas (primera fila o definida por configuración).
- **rows**: cada fila tiene `index` (número de fila) y `cells` como array de `{ key, value }` donde `key` es el header y `value` es un primitivo (`RawSheetCellValue`).
- **sheets**: el Worker devuelve **`RawParseResult`** con `sheets: Record<string, RawSheet>`. En XLSX/XLS puede haber varias hojas; en CSV una sola entrada. En v1 la UI usa la primera hoja; el contrato evita refactorizar cuando el usuario pida "Hoja 2".

### Output y entrega al flujo (Convert → Sanitizer → Validator)

El output del parser es la **entrada** del siguiente paso: **4. Convert** (alineación de headers al layout). Luego **5. Sanitizer** → **6. Validator** → **7. Transform** producen la **sheet** del sistema (ver **2. Setting.md** y **Architecture.md**).

- **Lo que entrega el parser**: un **RawParseResult** (`sheets: Record<string, RawSheet>`). Para el flujo actual se toma la hoja elegida (p. ej. la primera); cada **RawSheet** incluye **documentHash** (hash del documento para identificación única). Ese valor se guarda en el estado del Setting (`rawData`) y es el insumo de **Convert** y luego del resto del pipeline.
- **Lo que hace el pipeline** (Convert → Sanitizer → Validator → Transform): Convert alinea headers al layout; Sanitizer normaliza y castea; Validator compara con **fields** del **sheetLayout** y produce la estructura **sheet** (sistema general) definida en 2. Setting:
  - **sheet**: `name`, `sheetLayout`, `filesize`, `mimetype`, `extension`, `errors` (nivel hoja), `rows` con `row` → `{ index, errors, cells }` y **cell** → `{ key, value, errors }`.
- **Resultado para el usuario**: el flujo parser → validator genera un **artefacto/archivo** (o estructura en estado) con todo **distribuido y parseado**: datos ya mapeados a los fields del layout, con errores por hoja/fila/celda donde corresponda, listo para que lo uses en tu aplicación (guardado, exportación, o siguiente paso de negocio).

En resumen: el parser entrega RawSheet; el Validator lo confronta con los fields del sheetLayout y produce el **sheet** del Setting; ese sheet (o un archivo/export derivado) es lo que queda “todo distribuido y parseado” para uso final.

## Flujo en dos fases

### Fase 1 — Preview (primeras 10 filas)

1. Usuario selecciona/sube archivo.
2. Se envía el **Blob** al Web Worker una sola vez: **`worker.load(blob, options)`**.
3. El worker guarda el Blob (o stream/estado) en memoria interna y usa el **adapter** según extensión/MIME para parsear **solo las primeras 10 filas** (más la fila de cabecera si aplica).
4. Se devuelve un **RawParseResult** con `sheets` donde cada hoja tiene `rows` limitado a 10 filas y metadatos (name, filesize, headersCount, headers). Para CSV, incluir **encoding detectado** y **delimitador detectado** en la respuesta de preview (p. ej. en metadatos de la hoja o en un campo `parserMeta`) para que el usuario pueda corregir si ve caracteres extraños (p. ej. "Ã±").
5. Esta información se usa para:
   - **Mapeo de columnas**: el usuario puede asociar headers del archivo a campos del modelo.
   - **Validación de formato**: comprobar que el archivo tiene la estructura esperada antes de importar todo.
   - **Corrección de encoding/delimitador**: si el preview muestra caracteres incorrectos, el usuario puede indicar encoding o `delimiterOverride` y repetir preview.

### Fase 2 — Importación completa (si el formato aprueba)

1. Tras validar el mapeo/formato, **no se reenvía el Blob**. El worker **ya tiene el archivo cargado** desde la fase 1.
2. Se llama **`worker.parseAll(onProgress?)`**. El worker procesa el resto del archivo (o todo si usa streams/chunks) y devuelve **RawParseResult** completo.
3. Se notifica al usuario (éxito/error) y se entregan los datos para el siguiente paso del flujo.

**Persistencia en Worker**: Mantener el archivo/Workbook en el Worker entre Preview e Importación completa evita "double parsing" y re-envío de datos (crítico para archivos de 50MB+). La API del worker queda: `load(blob, options)` → preview + metadatos; `parseAll(onProgress?)` → resultado completo.

La comunicación con el usuario (estados: subiendo, preview listo, validando, importando, error) se puede centralizar en un **Provider** (React Context) que el parser y los componentes consuman.

## Integración con el Setting (paso 2)

El parser **consume** el provider y el estado definidos en **2. Setting.md**; no duplica estado global.

### Consumo del provider y estado

- **Provider**: el parser (y sus hooks) se usan **dentro** de `ImportProvider` y consumen el contexto vía **`useImporter()`**.
- **Estado consumido**: del `ImporterState` se lee `file` (archivo seleccionado) y se usa como origen para el Blob que se envía al worker.
- **Estado actualizado**: el parser actualiza el estado del Setting cuando corresponde:
  - Al iniciar lectura/parseo: actualizar `status` (p. ej. `'loading'`).
  - Al terminar preview o importación completa: actualizar `rawData` con el RawParseResult o la hoja elegida (según lo que el Setting espere para datos crudos) y `status` (`'success'` o `'error'`).
  - En caso de error: actualizar `status` a `'error'` y comunicar el mensaje (según cómo el Setting exponga errores).
- **Progreso**: no se escribe en el Context; se emite vía **EventTarget** (evento `importer-progress`). Solo la UI de progreso se suscribe. Ver *Progreso: EventTarget* más abajo.

Los hooks del parser (ej. `useImportSheet`) usarán `useImporter()` para leer `file` y para invocar los setters que actualicen `rawData` y `status`. Para progreso, el hook recibirá el callback del Worker y **disparará** el evento en el EventTarget del importer en lugar de actualizar estado en el Context.

### Progreso: EventTarget y throttle

Además del estado global del Setting, el parser mantiene y expone un **progreso local interno** del paso de parseo, para feedback en UI (p. ej. barra de progreso o mensaje “Importando…”).

- El Worker envía progreso por callback. En el main thread se debe pasar el callback envuelto con **`Comlink.proxy(onProgress)`** para que Comlink lo serialice correctamente. La capa que recibe ese callback **dispara un evento** en el **EventTarget** del importer (`importer-progress` con `detail: { phase, globalPercent, localPercent, currentRow, totalRows }`).
- **Throttle en el Worker**: si el Worker envía actualizaciones demasiado rápido (p. ej. cada fila en un archivo de 100k filas), se satura el Event Loop del Main Thread. **Estrategia**: implementar throttle en el Worker para enviar progreso solo cada **~100ms** o cada **~1% de avance** (lo que ocurra antes).
- **Solo** el componente de barra de progreso se suscribe y actualiza por **ref** o **useState** local; el resto de la UI no se re-renderiza.
- **Progreso por pasos**: si el flujo del parser tiene varios pasos (p. ej. “detectando formato” → “leyendo cabecera” → “cargando filas”), se puede exponer algo como `stepIndex` y `stepsTotal` (o “paso N de M”) y actualizarlo al avanzar.
- **Progreso por líneas**: cuando se usa streaming/chunks o se conoce el total de filas, se actualiza un progreso basado en **líneas cargadas vs total** (p. ej. `rowsLoaded / rowsCount`, o “X de Y líneas”). El worker envía mensajes de progreso ya throttled (`rowsProcessed`, `rowsTotal`) y el hook dispara el evento en el EventTarget.
- Este progreso es **local al step del parser** (no reemplaza `ImporterState`): puede vivir en el hook (ej. `useImportSheet`) como `parserProgress: { stepIndex, stepsTotal }` o `{ rowsLoaded, rowsCount }`, y la UI lo usa para mostrar el avance de “este paso” mientras `status` en el Setting sigue siendo `'loading'` hasta que termine todo.

## Arquitectura del parser

### 1. Universal Parser Adapter

- **Entrada**: `Blob` + opciones (p. ej. `maxRows?: number` para preview).
- **Detección**: por extensión del nombre del archivo o por `blob.type` (MIME).
- **Derivación**:
  - Si es `.xlsx`, `.xls`, `.ods` o MIME de Excel/ODS → **parser XLSX** (SheetJS).
  - Si es `.csv` o `text/csv` → **parser CSV** (Papa Parse).
- **Salida**: siempre **RawParseResult** (`sheets: Record<string, RawSheet>`) en el formato definido arriba (o error tipado).

### 2. Engines de parsing (xlsx, csv)

- **XLSX engine**: usa SheetJS para leer el workbook; puede devolver múltiples hojas. Mapea cada hoja a `RawSheet` (name, filesize, headers, rows con `index` y `cells` con `value: RawSheetCellValue`). En v1 se usa la primera hoja; el contrato expone `sheets: Record<string, RawSheet>`.
- **CSV engine**: usa Papa Parse con cabecera; normaliza a la misma estructura (una sola hoja en `sheets`). **Delimitador**: detección automática de `,` o `;`; opciones del Parser con **`delimiterOverride`**. **Encoding**: el Preview debe incluir el **encoding detectado** (p. ej. en metadatos) para que el usuario pueda corregir si ve caracteres incorrectos.

### 3. Web Worker (Comlink)

- **Responsabilidad**: exponer una API en dos fases. **Fase 1**: `load(blob, options?)` — recibe el Blob, lo mantiene en el Worker (o estado/stream listo), calcula **documentHash** (ver abajo), parsea preview y devuelve **RawParseResult** (sheets con pocas filas + metadatos, y para CSV: encoding/delimitador detectados). **Fase 2**: `parseAll(onProgress?)` — sin re-recibir el Blob, el worker termina de parsear y devuelve **RawParseResult** completo; para archivos **> 10MB** usar streams/chunks y reportar progreso vía callback (con throttle).
- **documentHash por streaming**: para calcular SHA-256 del Blob **no** se debe leer el Blob completo como un solo `ArrayBuffer` en archivos grandes (>100MB), ya que se duplicaría la memoria y el Worker podría fallar. **Enfoque**: leer el Blob en **trozos (chunks)** (p. ej. con `blob.slice()` o `blob.stream()` + reader) y actualizar el hash de forma incremental (Web Crypto permite `SubtleCrypto.importKey` + `sign`/digest por bloques, o usar una implementación de SHA-256 que acepte chunks si el entorno no expone API de streaming nativa). Así el Worker evita picos de memoria antes de empezar el parseo.
- **Comlink**: en el worker se usa **`Comlink.expose(api)`** para exponer `api = { load(blob, options?), parseAll(onProgress?) }`. En el main thread **`Comlink.wrap(worker)`**; para progreso, pasar el callback con **`Comlink.proxy(onProgress)`** en `parseAll`. El worker invoca el callback con throttling (~100ms o ~1% de avance).
- El worker importa el adapter y los engines; el bundler (ej. Vite/tsup) debe generar un entry point separado para el worker. No se usa un protocolo manual de mensajes; Comlink gestiona la comunicación.

### 4. Archivos grandes (streams/chunks, > 10MB)

- Si `filesize > 10 * 1024 * 1024` (10MB), no cargar todo en memoria: usar **streams** (e.g. `blob.stream()`) o procesamiento por **chunks**.
- El adapter/parsers deben soportar modo "streaming": leer por fragmentos, ir emitiendo filas o chunks de `RawSheetRow[]`, y al final construir o exponer el `RawSheet` (o un iterador/async generator según diseño).
- CSV: Papa Parse tiene modo stream; XLSX: evaluar lectura por chunks o streaming si la librería lo permite, o advertir límite para binarios muy grandes.

### 5. Encoding y delimitador (CSV)

- **UTF-8** por defecto; si se detecta que el contenido no es UTF-8 válido, intentar **ISO-8859-1** (Latin-1). No confiar solo en el intento manual: el Parser debe **devolver el encoding detectado** en la fase de Preview (p. ej. en metadatos de la hoja o en `parserMeta`) para que el usuario pueda corregirlo si ve caracteres extraños (p. ej. "Ã±").
- Las opciones del Parser deben aceptar **`delimiterOverride`** (y opcionalmente `encodingOverride`) para casos donde la detección automática de Papa Parse falle con CSV mal formados o encodings raros.
- Configurar o detectar encoding antes de pasar el texto a Papa Parse (p. ej. `FileReader` con encoding o librería de detección; o opciones de Papa si las soporta).

### 6. Comportamiento como hook y carpeta `hooks`

- Toda la funcionalidad del parser se expone como **un hook** (o familia de hooks) desde la carpeta del parser. Dentro de esa carpeta existirá la subcarpeta **`hooks/`** con los hooks necesarios (p. ej. `useParserWorker`, `usePreview`, `useImportSheet`).
- El "parser" en conjunto **se comporta como un hook**: el usuario usa algo como `useImportSheet()` y obtiene estado, preview, importación completa y comunicación con el worker sin tocar directamente el worker ni el adapter.
- El Provider de estado (comunicación con el usuario) puede vivir dentro de la misma carpeta y ser consumido por el hook principal.

### 7. Provider de estado y consumo del Setting

- El parser **consume** `ImportProvider` y `useImporter()` del **2. Setting.md**: lee `file`, actualiza `rawData` y `status` en el estado global.
- A nivel local del step, el hook/parser mantiene: fase (idle | previewing | validating | importing | error), resultado de preview (`RawSheet` con 10 filas), resultado de importación completa (`RawSheet`), mensajes de error. **Progreso** no se guarda en estado: se emite vía **EventTarget** (`importer-progress`); solo la UI de la barra de progreso se suscribe.
- Expone: métodos para iniciar preview, confirmar mapeo y lanzar importación completa, y reset; y el EventTarget (o acceso a él) para que la barra de progreso escuche el evento.
- Los componentes hijos consumen el hook y el contexto del Setting para mostrar mensajes, tabla de preview, barra de progreso (suscrita al evento) y resultado final.

## Plan de tareas (orden sugerido)

### Fase A — Tipos y contrato

1. Definir en TypeScript las interfaces `RawSheet`, `RawSheetRow`, `RawSheetCell`, el tipo **`RawSheetCellValue`** (`string | number | boolean | Date | null`), **`RawParseResult`** (`sheets: Record<string, RawSheet>`) y tipos de error si aplica, en un módulo compartido (ej. `src/types/sheet.ts`).

### Fase B — Engines de parsing (xlsx, csv)

2. **Engine XLSX** (SheetJS):
   - Instalar `xlsx`.
   - Implementar función que lea workbook y devuelva **RawParseResult** (todas las hojas mapeadas a `RawSheet`; en v1 se usa la primera). Opciones: `maxRows?`. Valores de celdas como **RawSheetCellValue**. Para archivos &gt; 10MB usar streams/chunks si la librería lo permite.

3. **Engine CSV** (Papa Parse):
   - Instalar `papaparse` y `@types/papaparse` si aplica.
   - Implementar función que devuelva **RawParseResult** (una sola hoja). Opciones: `maxRows?`, **`delimiterOverride`**, **`encodingOverride`**.
   - Encoding: forzar **UTF-8** o detectar **ISO-8859-1**; **devolver el encoding detectado** en metadatos del preview.
   - Delimitador: **detección automática** de `,` o `;`; si se pasa `delimiterOverride`, usarlo.
   - Para archivos &gt; 10MB usar modo stream de Papa Parse.

### Fase C — Adapter universal

4. **Adapter** (orquestador):
   - Crear `parseSheet(blob, options?)` que devuelva **Promise&lt;RawParseResult&gt;** (o equivalente para uso interno con `load` + `parseAll`). Opciones: `maxRows?`, `fileName?`, `delimiterOverride?`, `encodingOverride?`.
   - Obtener extensión (nombre del archivo en el Blob o en `options.fileName`) o `blob.type`.
   - Mapa extensión/MIME → engine XLSX o CSV (carpeta `engines/`); si no hay coincidencia, rechazar con error claro.
   - Si `blob.size > 10MB`, usar flujo por streams/chunks internamente.

### Fase D — Web Worker (Comlink)

5. **Worker**:
   - Instalar **Comlink** y crear el entry del worker (ej. `parser.worker.ts`). Exponer un objeto API con **`Comlink.expose(api)`** con **`load(blob, options?)`** y **`parseAll(onProgress?)`**. **load**: recibe el Blob, lo mantiene en el Worker, calcula documentHash por streaming (chunks), parsea preview y devuelve `Promise<RawParseResult>`. **parseAll**: sin re-recibir el Blob, termina de parsear y devuelve `Promise<RawParseResult>`; progreso vía **`Comlink.proxy(onProgress)`** con throttle (~100ms o ~1%).
   - Configurar bundler para generar el worker. El wrapper en `hooks/` expone `load()` y `parseAll()` y devuelve RawParseResult.
   - **Ciclo de vida:** El hook que crea el Worker debe guardar la instancia en una ref (o pasarla al Provider/worker manager) y en el **cleanup del useEffect** ejecutar `worker.terminate()`. Así `abort()` y el desmontaje del componente terminan el Worker y liberan recursos.

### Fase E — Preview (primeras 10 filas)

6. Primera llamada: **`worker.load(blob, { maxRows: 10, ... })`**. Respuesta = preview (RawParseResult con pocas filas y metadatos; para CSV, encoding/delimitador detectados). Importación completa con **`worker.parseAll(onProgress?)`** sin re-enviar el Blob.
   - Documentar que la respuesta de load es el “preview” para mapeo y validación.

### Fase F — Provider y comunicación con el usuario

7. **Hooks, consumo del Setting y progreso** (carpeta del parser con `hooks/`):
   - Integrar con **2. Setting**: los hooks del parser usan `useImporter()` para leer `file` y actualizar `rawData` y `status` cuando termina preview o importación completa.
   - Crear en `hooks/` los hooks necesarios: `useParserWorker`, `usePreview`, y el hook principal (ej. `useImportSheet`) que orquesta estado, preview e importación completa.
   - Estado local del step: fase (idle | previewing | importing | error), preview (RawParseResult con pocas filas), resultado completo (RawParseResult), error. Para el Setting se toma la hoja elegida (p. ej. la primera de `sheets`). **Progreso**: no en estado; cuando el worker envíe avance (callback), el hook **dispara** el evento `importer-progress` en el EventTarget del importer; solo la barra de progreso se suscribe a ese evento.
   - Acciones: subir archivo e iniciar preview, confirmar mapeo/validación e importar todo, reset. Todo expuesto a través del hook; los componentes consumen el hook y el contexto del Setting; la barra de progreso consume el EventTarget para evitar re-renders masivos.

### Fase G — Integración y tests unitarios

**Se deben crear tests unitarios para todo este proceso** con Vitest (ver también **2. Setting.md** — Tests unitarios en el plan general). Alcance mínimo:

8. **Tests unitarios del parser**:
   - **Engines**: XLSX y CSV con fixtures; comprobar RawParseResult, RawSheet (headers, rows, cells con RawSheetCellValue), opción `maxRows`, y para CSV `delimiterOverride`/encoding detectado.
   - **Adapter**: derivación por extensión/MIME a engines; salida RawParseResult unificada; rechazo para tipos no soportados.
   - **documentHash**: que cada RawSheet incluya `documentHash` (calculado por streaming/chunks) y que el mismo Blob produzca el mismo hash.
   - **Worker**: mock de Blob; comprobar `load()` devuelve preview y `parseAll()` devuelve RawParseResult completo sin re-enviar Blob; formato de respuesta y throttle de progreso.
   - **Hooks** (opcional según complejidad): que `useImportSheet` / `useParserWorker` actualicen estado y progreso según lo esperado; se puede usar `@testing-library/react` y mock del Setting.
9. **Integración**: pantalla mínima (o ejemplo en docs/storybook): input file → preview 10 filas → botón “Importar todo” → mostrar resultado completo (o usar el Provider en un ejemplo en docs o storybook).

**Criterio de éxito**: suite de tests Vitest del módulo Parser en verde antes de cerrar el paso.

## Estructura de carpetas sugerida

El parser se organiza en una carpeta que **crea y usa `hooks/`**; toda la API pública se comporta como hook(s).

```
src/
  types/
    sheet.ts              # RawSheet, RawSheetRow, RawSheetCell, RawSheetCellValue, RawParseResult
  parser/                 # módulo del parser (se comporta como hook)
    adapter.ts            # orquestador: parseSheet (universal), delega a engines
    engines/              # estrategias de lectura (no confundir con adapter = formateador unificado)
      xlsx-parser.ts      # engine XLSX (SheetJS)
      csv-parser.ts       # engine CSV (Papa Parse: encoding, delimiterOverride)
    worker/
      parser.worker.ts    # entry del worker (load + parseAll; documentHash por streaming; streams/chunks si > 10MB)
    hooks/
      useParserWorker.ts  # load() y parseAll(), devuelve RawParseResult
      usePreview.ts       # preview (primeras filas)
      useImportSheet.ts   # hook principal: estado + preview + importación completa
    ImportSheetProvider.tsx
    types.ts              # estado y acciones del provider
  index.ts                # re-export público (hook principal, tipos, provider)
```

## Criterios de aceptación (resumen)

- [x] Adapter deriva correctamente por extensión y MIME a XLSX o CSV.
- [x] Salida siempre cumple el contrato **RawParseResult** (`sheets: Record<string, RawSheet>`) y **RawSheetCellValue** (primitivos); cada RawSheet con name, filesize, documentHash, rowsCount, headersCount, headers, rows.
- [x] RawSheet incluye **documentHash** calculado por **streaming/chunks** (no leer el Blob completo en memoria) para identificación única.
- [x] Worker vía **Comlink** con API en dos fases: **load(blob, options?)** (mantiene Blob en Worker, devuelve preview) y **parseAll(onProgress?)** (sin re-enviar Blob); salida `Promise<RawParseResult>`.

## Notas

- **Dependencia del Setting**: el parser asume que **2. Setting** está implementado (ImportProvider, useImporter, ImporterState). Se ejecuta dentro del provider y actualiza `rawData` y `status`; además expone progreso local del step (pasos o líneas).
- **Output y pipeline**: el parser entrega **RawParseResult** (`sheets`); **Convert** alinea headers; **Sanitizer** y **Validator** consumen la hoja y generan el tipo **sheet** del Setting; **Transform** aplica transformaciones. Ese sheet (o archivo derivado) es el resultado "todo distribuido y parseado" para la app.
- **Nombre del archivo en Blob**: en el browser, el `File` tiene `name`; si solo recibes un `Blob`, pasar el nombre por opciones (`fileName`) para extensión y para `RawSheet.name` en CSV.
- **Múltiples hojas**: el contrato **RawParseResult** con `sheets: Record<string, RawSheet>` prevee múltiples hojas desde día 1; en v1 la UI usa la primera hoja; evita refactorizar cuando el usuario pida "Hoja 2".
- **Encoding CSV**: forzar **UTF-8** o detectar **ISO-8859-1**; devolver encoding detectado en Preview; opción **encodingOverride**. **Delimitador**: detección automática; opción **delimiterOverride**.
- **documentHash**: calcular en el Worker por **streaming/chunks** (no leer el Blob completo como un solo ArrayBuffer en archivos grandes); mismo archivo → mismo hash en hex.

Cuando este plan esté validado, el siguiente paso puede ser la Fase A (tipos) y luego B–C (engines y adapter), dejando Worker y hooks/Provider para después de tener el adapter estable.
