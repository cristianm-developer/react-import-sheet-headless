# 5. Sanitizer — Plan de construcción

**Status:** Implemented (2026-02-27). SanitizedSheet in state; runner + worker + useSanitizerWorker; Convert → Sanitizer → Validator flow and docs in place.

## Objetivo

Añadir una **capa de sanitización** que reciba el resultado del parser (`RawSheet`), ejecute sanitizers **síncronos** por nivel (celda, fila, hoja) y devuelva un **SanitizedSheet** con valores ya limpiados, tipados (casting por `valueType`) y opcionalmente con filas descartadas. Esta capa se ejecuta **antes** del Validator (6. Validator); en lugar de añadir errores **normaliza, hace casting y limpia** los datos. Todos los sanitizers son **síncronos** para maximizar velocidad en el Worker y evitar overhead de promesas.

---

## Contexto en el flujo (2. Setting y 3. Parser)

- **2. Setting.md** define los tipos compartidos y que el parser entrega datos crudos (`rawData` / `RawSheet`) para el resto del flujo.
- **3. Parser.md** entrega **RawSheet** (RawSheetCell: `key`/`value`, RawSheetRow: `index` + `cells`, RawSheet: `name`, `filesize`, `documentHash`, `headers`, `rows`, etc.).
- El **Sanitizer** consume el output de **Convert** (**ConvertedSheet**) y produce un **SanitizedSheet** (valores con casting por `valueType`, filas descartables, sin errores) que será la **entrada estricta del Validator** (6. Validator).

Flujo: **Parser → Convert → Sanitizer → Validator → Transform** → sheet (sistema general).

---

## Entrada y salida

| Concepto | Origen | Descripción |
|----------|--------|-------------|
| **Entrada** | Convert (4. Convert) + Setting | **ConvertedSheet** (output de Convert) y configuración de sanitizers en `sheetLayout` (y `valueType` por campo para casting). |
| **Salida** | Tipo dedicado | **SanitizedSheet**: misma estructura que RawSheet pero con **valores ya casteados** según `valueType` del layout, filas descartadas eliminadas (`rows` y `rowsCount` actualizados) y valores normalizados por los sanitizers. No se añaden `errors`; los sanitizers solo transforman/normalizan o descartan filas. |

El Validator (paso 4) recibirá siempre el **output del Sanitizer** con firma estricta: `validate(data: SanitizedSheet)`, no el ConvertedSheet ni el RawSheet crudo.

---

## Diferencia con Validator

| Aspecto | Sanitizer | Validator |
|---------|-----------|-----------|
| Propósito | Casting por `valueType`, limpiar y normalizar valores (trim, formato, vacíos), descartar filas inválidas | Validar reglas y rellenar `errors` en cell/row/sheet |
| Salida | **SanitizedSheet** con `value` tipados y datos actualizados | **sheet** (tipos del Setting) con `errors` por nivel |
| Errores | No añade errores | Añade errores a `cell.errors`, `row.errors`, `sheet.errors` |
| Orden en el flujo | Se ejecuta **antes** del Validator (paso 4) | Se ejecuta **después** del Sanitizer (paso 3) |

Los sanitizers tienen la misma **forma de ejecución** que los validators (por fila: celdas → fila; al final hoja). **Cell and Row sanitizers MUST be synchronous** para rendimiento; si en el futuro se permiten sanitizers de nivel **table** async (p. ej. normalización que llame al backend), deben seguir las mismas reglas de resiliencia que los table validators (try/catch, código de error tipo `EXTERNAL_*`, AbortController). Por ahora el contrato es **entrada/salida de datos limpiados** (o descarte de fila), no de errores.

## Configuración de sanitizers (sheetLayout o equivalente)

En **2. Setting.md** el `sheetLayout` define validators, transformations y **valueType** por campo. Para el Sanitizer se añade configuración de **sanitizers** por nivel; **todos son síncronos**:

- **sheet-sanitizers** `[]`: `(sheet: ConvertedSheet) => ConvertedSheet` — reciben la hoja y devuelven la hoja con valores/metadatos normalizados (pueden mutar en el Worker).
- **row-sanitizers** `[]`: `(row: ConvertedSheetRow) => ConvertedSheetRow | null` — reciben la fila y devuelven la fila normalizada, o **`null`** para **descartar la fila** (no llegará al Validator; se actualiza `rows` y `rowsCount`).
- **field.sanitizers** `[]` (por campo/celda): `(cell: RawSheetCell, row: ConvertedSheetRow) => RawSheetCell` — reciben la celda **y la fila completa** (contexto para normalizaciones que dependen de otras celdas, p. ej. "ID de Empleado" según "País") y devuelven la celda con `value` (y si aplica `key`) normalizados (pueden mutar en el Worker).

Cada sanitizer es **síncrono**. La ejecución es secuencial en el Worker, sin promesas, para máximo rendimiento. La implementación puede vivir en `src/types/sheet-layout.ts` extendiendo `sheetLayout` con `sheetSanitizers`, `rowSanitizers` y en cada field `sanitizers`.

**Ubicación de implementaciones:** Las funciones sanitizer reutilizables viven en **`src/utils/controller/[context]/`**, agrupadas por campo/contexto (p. ej. `email`). Nomenclatura: `cell-{context}-sanitizer.ts`, `row-{context}-sanitizer.ts`, `table-{context}-sanitizer.ts` (p. ej. `cell-email-sanitizer.ts`). El layout referencia por nombre; el worker o el runner resuelven desde ese controller (ver Architecture.md — Utils).

**Tipo en el Registry:** Cada entrada del Registry de sanitizers debe llevar **`type: 'cell' | 'row' | 'table'`** para que el Runner invoque **cell/row** por cada fila/celda y **table** una sola vez al final del ciclo (con acceso al dataset completo). Ver **Architecture.md** — *Registry entry type*.

**Patrón Register() (built-in y custom):** Cada sanitizer built-in (cell, row, sheet) expone **`Register(registerSanitizer?)`** que devuelve el identificador string y opcionalmente registra la implementación en el Provider/Worker si se pasa el callback. El layout puede usar `sanitizers: [CellEmailSanitizer.Register(registerSanitizer), 'trim']`. La librería expone interfaces **`CellSanitizerDefinition`**, **`RowSanitizerDefinition`**, **`SheetSanitizerDefinition`** (en `src/types/` o en el paso 2) para que el usuario cree sus propios sanitizers con el mismo contrato y los use en el layout. Ver **Architecture.md** — *Register() pattern* y **2. Setting** — *Patrón Register()*.

**Funciones puras y params:** Si se usa serialización al Worker (`fn.toString()`), los sanitizers deben ser puros. Cualquier configuración (listas, opciones) debe pasarse por **params** en el layout (ej. `{ name: 'normalize-with-map', params: { map: { 'a': 'A' } } }`). Ver **Architecture.md** — *Technical note: Serialization to Worker and pure functions*.

## Casting explícito por valueType

Antes de ejecutar los sanitizers personalizados del usuario, el Sanitizer aplica un **paso de casting automático** basado en el `valueType` del `sheetLayout` por campo. Si el layout indica que una columna es `number`, el Sanitizer convierte el `unknown` del Parser en un `number` real (o `null`/valor por defecto si no es convertible). Así el Validator recibe un **SanitizedSheet** con tipos concretos y no tiene que lidiar con `unknown`.

Orden: **casting por valueType → field.sanitizers (celda) → row-sanitizers → sheet-sanitizers.**

---

## Orden de ejecución (secuencial, síncrono)

Idéntico al Validator en cuanto al orden; todo síncrono:

1. **Por cada fila (row)** del `RawSheet`:
   - **Por cada celda (cell)** de la fila: se resuelve el **field** por `cell.key`. Primero se aplica el **casting** según `field.valueType`. Luego se ejecutan en secuencia los **field.sanitizers**; cada uno recibe `(cell, row)` y devuelve la celda con `value` (y si aplica `key`) ya normalizados. Dentro del Worker se permite **mutación controlada** del objeto (evitar crear nuevos objetos por celda para no saturar al GC en archivos grandes).
   - Tras procesar todas las celdas, se ejecutan los **row-sanitizers**. Si alguno devuelve **`null`**, la fila se **descarta** (no se añade al resultado; se actualizará `rowsCount`).
   - Se pasa a la siguiente fila y se repite.

2. **Al terminar todas las filas**: se ejecutan los **sheet-sanitizers** sobre el objeto hoja (con `rows` ya filtrados) y se devuelve el **SanitizedSheet** final (entrada del Validator).

Resumen: **por fila → casting por valueType → cell-sanitizers (cell, row) → row-sanitizers (null = descartar fila) → siguiente fila; al final → sheet-sanitizers. Salida: SanitizedSheet.**

## Web Worker y progreso (Comlink + EventTarget)

- La lógica de sanitización se ejecuta en un **Web Worker** (igual que el Validator) para no bloquear la UI. La comunicación con el worker se hace con **Comlink**. La capa que invoca al Worker (main thread) debe aplicar **timeout y recuperación** como en **Architecture.md** — *Maturity & resilience → Worker timeout & recovery*.
- **Abort y ciclo de vida:** El hook que instancia el Worker (p. ej. `useSanitizerWorker`) debe guardar la instancia en una **ref** accesible para el Provider o para **`abort()`**; en el **cleanup de `useEffect`** al desmontar debe llamar **`worker.terminate()`**. Ver **Architecture.md** — *Process Abort & Worker Cleanup* y **2. Setting** — *Abort y ciclo de vida*.
- En el worker: **`Comlink.expose(api)`** con un método (p. ej. `sanitize(rawSheet, sheetLayout, options?, onProgress?)`) que ejecute `runSanitization` **síncronamente** (sanitizers sync) y devuelva `Promise<SanitizedSheet>`. El progreso se reporta al main thread con **`Comlink.proxy(onProgress)`**.
- **Throttling del progreso**: para no saturar el hilo principal, **no** se emite un evento por cada fila procesada. Se aplica throttling: como máximo un evento cada **16 ms** (~60 fps) o cada **1 % de progreso**, lo que ocurra antes. Igual criterio que en el Parser (ver Architecture.md).
- En el main thread: **`Comlink.wrap(worker)`** para obtener un proxy; al recibir el callback de progreso (ya throttled), **no** actualizar el Context. En su lugar, **disparar un evento** en el **EventTarget** del importer (p. ej. `importer-progress` con `detail: { phase, globalPercent, localPercent, currentRow, totalRows }`). Solo la UI de la barra de progreso se suscribe a ese evento (evitar re-renders masivos). Ver **2. Setting.md** y **Architecture.md** — *Progress and re-renders: EventTarget*.
- **Mutación en el Worker**: dentro del Worker el ConvertedSheet es efímero y el entorno está aislado. Se permite **mutación controlada** de celdas/filas/hoja para evitar miles de asignaciones y presión al GC; los sanitizers pueden modificar el objeto que reciben en lugar de devolver copias nuevas.
- **Estrategia delta (opcional)**: para reducir tráfico Worker ↔ main, el Worker puede devolver solo **errores** o **deltas** y el main parchear su copia local; ver **Architecture.md** — *Data efficiency: Delta strategy*. Para el Sanitizer la salida es SanitizedSheet; la delta aplica sobre todo al Validator/Transform.

---

## Alcance funcional (resumen)

- **Entrada**: **ConvertedSheet** (de Convert) + configuración de sanitizers (sheet, row, field/cell) y `valueType` por campo (para casting).
- **Ejecución**: mismo orden que el Validator (por row → casting por valueType → cell-sanitizers(cell, row) → row-sanitizers; al final sheet-sanitizers). **Todo síncrono** y secuencial; row-sanitizer puede devolver `null` para descartar la fila.
- **Salida**: **SanitizedSheet** con valores casteados y normalizados; filas descartadas eliminadas (`rows` y `rowsCount` actualizados); sin añadir errores.
- **Integración**: el output del Sanitizer es la **entrada estricta del Validator** (`validate(data: SanitizedSheet)`); el flujo parser → sanitizer → validator queda fijado en el diseño.

## Plan de tareas (orden sugerido)

### Fase A — Tipos y contrato

1. **Extender tipos en `src/types`**  
   - Añadir en `sheetLayout` las propiedades opcionales: `sheetSanitizers`, `rowSanitizers`, y en cada field `sanitizers` (firmas síncronas: sheet/row/cell; row puede devolver `null`; cell recibe `(cell, row)`). Definir tipo **SanitizedSheet** (estructura como ConvertedSheet pero con valores ya tipados según `valueType` y filas descartadas eliminadas). Documentar en 2. Setting que el flujo es Parser → Convert → Sanitizer → Validator y que el Validator recibe `SanitizedSheet`.

2. **Tipos de mensajes del worker (sanitizer)**  
   - Definir mensajes de entrada (ConvertedSheet + config de sanitizers o identificadores) y salida: progreso (throttled), éxito (**SanitizedSheet**), error.

### Fase B — Runners de sanitización

3. **Casting por valueType (applyValueTypeCasting o equivalente)**  
   - Por cada celda, según el `valueType` del field en el layout, convertir `value` de `unknown` al tipo indicado (number, string, date, boolean, etc.). Ejecutar **antes** de los sanitizers de celda.

4. **runCellSanitizers(cell, row, field, sheetLayout)**  
   - Ejecutar en secuencia los `field.sanitizers`; cada uno recibe `(cell, row)` y devuelve la celda con `value` (y si aplica `key`) normalizados. Permitir mutación en el Worker. Sin tocar errores (RawSheetCell no tiene `errors`).

5. **runRowSanitizers(row, sheetLayout)**  
   - Ejecutar en secuencia los `rowSanitizers` del layout; entrada `RawSheetRow`, salida `RawSheetRow | null`. Si algún sanitizer devuelve `null`, la fila se considera descartada.

6. **runSheetSanitizers(sheet, sheetLayout)**  
   - Ejecutar en secuencia los `sheetSanitizers`; entrada/salida tipo ConvertedSheet (en memoria puede mutar; resultado final es SanitizedSheet).

7. **runSanitization(convertedSheet, sheetLayout, onProgress?)**  
   - Bucle: por cada row → casting por valueType por celda → runCellSanitizers(cell, row, ...) → runRowSanitizers (si devuelve `null`, no añadir la fila al resultado y actualizar conteo); al final runSheetSanitizers. Llamar `onProgress` con **throttling** (máx. cada 16 ms o cada 1 % de progreso). Devolver **SanitizedSheet**.

### Fase C — Web Worker (Comlink)

8. **Estrategia de sanitizers en worker**  
   - Igual que en Validator: sanitizers por nombre/registro en el worker, o layout con identificadores y worker importando implementaciones. No se pueden pasar funciones por valor; Comlink serializa argumentos/retornos.

9. **sanitizer.worker.ts**  
   - Usar **Comlink**: exponer con **`Comlink.expose(api)`** un objeto con método `sanitize(convertedSheet, sheetLayout, options?, onProgress?)` que ejecute `runSanitization` de forma síncrona en el worker (invocando `onProgress` con throttling; el main pasará el callback con **`Comlink.proxy(onProgress)`**). Devolver `Promise<SanitizedSheet>`. Configurar entry del worker en el bundler.

### Fase D — Hook e integración con el flujo

10. **Hook useSanitizerWorker (o equivalente)**  
   - Instanciar el worker y obtener el proxy con **`Comlink.wrap(worker)`**. Exponer `sanitize(convertedSheet, sheetLayout, options?)` que llame al proxy (p. ej. `await sanitizerProxy.sanitize(...)`) y devuelva `Promise<SanitizedSheet>`. Para progreso: pasar callback con `Comlink.proxy(onProgress)` (throttled en worker); en el callback, **disparar evento** `importer-progress` en el EventTarget del importer (no actualizar Context). Solo la barra de progreso se suscribe al evento.

11. **Integración Convert → Sanitizer → Validator**  
    - Tras obtener **ConvertedSheet** (de Convert), invocar el Sanitizer; pasar el **SanitizedSheet** resultante al Validator. Ajustar provider/hooks del Setting para que el Validator reciba siempre el output del Sanitizer con firma `validate(data: SanitizedSheet)` (p. ej. guardando en estado `sanitizedData` o pasando directamente al validador sin persistir, según diseño).

### Fase E — Tests unitarios e integración

12. **Tests unitarios (Vitest)** del módulo Sanitizer: casting, runners, orquestador, worker y hook (alcance detallado en la sección *Tests unitarios (contexto Sanitizer)* más abajo).

13. **Test del worker** (o del runner sin worker en entorno de test) con un RawSheet pequeño y un sheetLayout con sanitizers simples; comprobar mensajes de progreso (throttling) y **SanitizedSheet** final con valores casteados y sanitizados; comprobar que row-sanitizer que devuelve `null` descarta la fila.

14. **Integración** en el flujo Convert → Sanitizer → Validator: tras obtener ConvertedSheet, invocar el Sanitizer y pasar el **SanitizedSheet** al Validator; verificar en UI o tests de integración que el Validator recibe el SanitizedSheet (firma estricta).

- **useSanitizerWorker** (o hook del sanitizer): que el estado (status) se actualice correctamente; que al recibir progreso del worker se dispare el evento `importer-progress` en el EventTarget (mock del EventTarget si aplica).

Los tests deben vivir junto al código del sanitizer (p. ej. `*.test.ts` / `*.spec.ts` en las mismas carpetas o en una carpeta `__tests__` del módulo sanitizer) y ejecutarse con Vitest. **Criterio de éxito**: suite de tests del módulo Sanitizer en verde antes de cerrar el paso.

- [ ] Entrada: **ConvertedSheet** (output de Convert) + configuración de sanitizers por nivel (sheet, row, field/cell) y `valueType` por campo en sheetLayout.

- [ ] El Validator recibe como entrada **SanitizedSheet** con firma `validate(data: SanitizedSheet)`.
- [ ] Tipos en Setting/sheetLayout extendidos con sanitizers y tipo **SanitizedSheet**; flujo Parser → Convert → Sanitizer → Validator documentado.
- [ ] **Tests unitarios** creados (casting, runCellSanitizers con (cell, row), runRowSanitizers con descarte `null`, runSheetSanitizers, runSanitization, worker, useSanitizerWorker); tests junto al código (`*.test.ts` / `__tests__`); suite Vitest del módulo Sanitizer en verde antes de cerrar el paso.
- [ ] **Patrón Register():** Cada sanitizer built-in (cell, row, sheet) exporta `id`, `type: 'cell'|'row'|'table'` y `Register(registerSanitizer?)`; al registrar se guarda **type** en el Registry para que el Runner invoque cell/row por fila y table una vez al final. Interfaces **CellSanitizerDefinition**, **RowSanitizerDefinition**, **SheetSanitizerDefinition** expuestas para sanitizers custom (ver Architecture y 2. Setting).
- [ ] **Abort y cleanup:** El hook guarda el Worker en ref; cleanup en desmontaje llama `worker.terminate()`; el Provider puede terminar este Worker vía `abort()`.

## Ejemplos de sanitizers típicos

- **Celda** `(cell, row) => cell`: trim de strings, convertir vacíos a `null` o `''`, normalizar números (coma → punto), normalizar fechas a ISO; o normalizar "ID de Empleado" según el valor de "País" en la misma `row`.
- **Fila** `(row) => RawSheetRow | null`: eliminar filas donde todas las celdas estén vacías (devolver `null` para descartar), normalizar orden de celdas; si la fila está corrupta o es basura, devolver `null` para que el Validator no la procese.
- **Hoja**: normalizar nombres de headers, eliminar filas completamente vacías (o ya filtradas por row-sanitizer), etc.

El **casting** por `valueType` se hace antes de los sanitizers de celda; la API genérica síncrona `(cell, row) => cell` y `(row) => row | null` permite cualquier normalización que no dependa de añadir errores.
