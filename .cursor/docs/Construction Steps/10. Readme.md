# 10. Readme — Plan para el README profesional

## Objetivo

Definir y ejecutar el plan para un **README profesional** que funcione como página de ventas del paquete: instalación clara, Quick Start con el hook, referencia a documentación de schema (validadores/transformadores en otro README) y contribución con Conventional Commits y tests obligatorios.

---

## READMEs distribuidos

Usaremos **READMEs distribuidos** para que la documentación viva junto al código y sea fácil de mantener:

| Ubicación | Propósito |
|-----------|-----------|
| **README principal** (`README.md` en la raíz) | Página de ventas del paquete: instalación, Quick Start, enlaces a docs, Schema Docs (resumen + enlaces), contribución. Ver *Estructura del README principal* más abajo. |
| **README de controllers** (`src/utils/controller/README.md`) | Describe **cómo** se organizan los controllers: dónde van validadores, transforms y sanitizers, en qué orden se ejecutan (cell → row → sheet) y **cómo se implementan** (naming `{level}-{context}-{type}.ts`, registro en layout, uso desde workers). Debe enlazar a [Architecture](.cursor/docs/Architecture.md) para la estructura de carpetas y el flujo del pipeline. |
| **README por controller** (p. ej. `src/utils/controller/email/README.md`) | Cada controller en particular (email, phone, date, etc.) puede tener su propio README que describa **qué hace** ese controller y **cómo usarlo**: qué sanitizers/validators/transforms expone, parámetros, ejemplos de uso en el layout y enlaces de vuelta al README de controllers. |

**Navegación:** El README principal enlaza a `docs/validators.md`, `docs/transformers.md` y opcionalmente a `src/utils/controller/README.md` (para “cómo añadir o usar controllers”). El README de controllers enlaza a Architecture y a cada controller que tenga README (email, etc.). Cada controller README enlaza de vuelta al README de controllers.

## Estructura del README principal (raíz del package)

El README en la **raíz** del repositorio debe incluir estas secciones en este orden:

### 1. Título y descripción breve

- **Título**: nombre del paquete (p. ej. `@cristianm/react-import-sheet-headless`).
- **Descripción**: 1–2 frases sobre qué hace la librería. **SEO/Discovery**: incluir las palabras clave **"React"**, **"Headless"**, **"Import"**, **"Excel"** (o CSV/sheet), **"Web Worker"** para mejorar descubrimiento en búsquedas.
- Opcional: badges (npm version, license, tests).

### 2. Instalación

- Una sola línea de comando, sin ambigüedad:
  ```bash
  npm install @cristianm/react-import-sheet-headless
  ```
- Mencionar **peer dependencies**: React 18+ (ya definido en `package.json`).

### 2. Por qué Headless

- **Objetivo**: justificar la decisión headless en una frase poderosa.
- **Frase sugerida**: *"Trae tus propios componentes, nosotros ponemos la lógica, la velocidad de los Web Workers y la validación masiva."* Deja claro por qué no hay un componente `<Table />` preconstruido.

### 3. Visualización del pipeline

- Aunque es headless, una **imagen o diagrama** del flujo de datos ayuda a entender qué ocurre "bajo el capó".
- Incluir (en README o en `docs/`) un esquema del flujo: **Input File → Parser (Worker) → Sanitizer (Worker) → Validator (Worker) → Transform (Worker) → Result (sheet) + Errors**. Opcional: edición de celdas sobre el resultado.
- Formato: imagen (PNG/SVG) o diagrama Mermaid en el README.

### 4. Quick Start (regla de oro: ~10 líneas)

- **Objetivo**: en **~10 segundos** el desarrollador debe entender el valor de la librería; si no, la abandonará. El ejemplo debe ser **puramente lógico** (headless), sin UI de tabla.
- **Ejemplo refinado** para el README principal:

```tsx
import { ImportProvider, useSheetView } from '@cristianm/react-import-sheet-headless';

function App() {
  return (
    <ImportProvider layout={myLayout}>
      <ImporterComponent />
    </ImportProvider>
  );
}

function ImporterComponent() {
  const { sheet, editCell, getPaginatedResult } = useSheetView({ pageSize: 10 });

  if (!sheet) return <input type="file" onChange={...} />;

  return (
    <div>
      <p>Rows: {sheet.rowsCount} | Errors: {sheet.errors.length}</p>
      {/* Tu componente de tabla aquí */}
    </div>
  );
}
```

- Código completo y ejecutable en formato markdown (bloque ` ```tsx `) para copiar y probar.
- **Regla**: corto; detalles avanzados (layout, validadores personalizados) se enlazan a la documentación de schema/validadores.

### 5. How to (usage)

- **Objetivo**: documentar **cómo usar** lo implementado (hooks, provider, validadores, transformadores, edición, etc.) para que cualquier usuario sepa usar las piezas sin leer el código.
- **Contenido**: enlazar desde el README principal (p. ej. "See [How to / Usage](docs/how-to.md) for step-by-step usage and recipes.").
- **Mantenimiento**: este contenido **debe actualizarse** cuando se añadan o cambien features. Ver regla en `.cursor/rules/typescript-standards.mdc`.

### 6. Schema Docs (referencia, no documentación completa)

- **Objetivo**: tabla clara de **qué opciones acepta** el sistema (validadores, transformadores, niveles).
- **En el README principal**:
  - **Solo listar** los conceptos y enlazar a documentación separada:
    - **Validadores**: tabla resumida (nombre/nivel: cell | row | sheet) y **enlace** a un README dedicado (p. ej. `docs/validators.md` o `docs/schema/validators.md`) donde estén **todos los validadores existentes** documentados (opciones, parámetros, ejemplos).
    - **Transformadores**: tabla resumida (nombre/nivel) y **enlace** a `docs/transformers.md` (o `docs/schema/transformers.md`) donde estén los transformadores existentes.
  - Opcional: tabla del **SheetLayout** (campos principales: `fields`, `sheet-validators`, `row-validators`, etc.) con una frase cada uno y enlace a `docs/schema.md` o equivalente para el contrato completo.
- **No** duplicar aquí la documentación detallada de cada validador/transformador; eso vive en los READMEs referenciados. Aquí solo: “qué hay” + “dónde ver los detalles”.

### 7. Contribución

- **Conventional Commits**: indicar que el proyecto usa [Conventional Commits](https://www.conventionalcommits.org/) para mensajes de commit (p. ej. `feat:`, `fix:`, `docs:`). Mencionar que los PRs que no sigan esta convención pueden ser rechazados o pedidos de ajuste.
- **Tests obligatorios**: dejar claro que:
  - Los tests (Vitest) son **obligatorios** para contribuir.
  - Antes de abrir un PR hay que ejecutar `npm run test` y asegurarse de que pasen.
  - Opcional: que el pre-push (Husky) ejecuta los tests y que la cobertura se mantiene (según 1. PackageSetting).
- **Efecto buscado**: “Ahuyenta a contribuidores mediocres y atrae a los buenos” — tono profesional y directo, sin sonar hostil.

### 8. Licencia y enlaces

- Licencia (MIT según `package.json`).
- Enlaces útiles: repo, issues, documentación adicional (docs/).

## Resumen de la sección “Schema Docs” en el README

En el README principal, algo en esta línea:

```markdown
## Schema Docs

El layout de la hoja (`SheetLayout`) define validadores y transformadores por nivel (cell, row, sheet). Las opciones que acepta cada uno se documentan en:

| Tipo           | Documentación |
|----------------|---------------|
| Validadores    | [Validadores existentes](docs/validators.md) |
| Transformadores| [Transformadores existentes](docs/transformers.md) |

Resumen por nivel:
- **Validadores**: cell (por campo), row, sheet — ver [docs/validators.md](docs/validators.md).
- **Transformadores**: cell, row, sheet — ver [docs/transformers.md](docs/transformers.md).
```

### Estructura de Schema Docs (validators): tabla de compatibilidad

En **docs/validators.md** no solo listar validadores; usar una **tabla de compatibilidad** (tip senior) para que el usuario encuentre rápido "cómo valido un CUIT" o "qué parámetros tiene min-length":

| Validador   | Nivel | Parámetros      | Ejemplo |
|-------------|-------|-----------------|---------|
| required    | Cell  | N/A             | `validators: ['required']` |
| min-length  | Cell  | `{ min: number }` | `validators: [{ name: 'min-length', params: { min: 5 } }]` |
| unique      | Sheet | `{ column: string }` | `sheetValidators: ['unique']` o con params según implementación |

En `docs/validators.md` y `docs/transformers.md` irán estas tablas detalladas más la documentación de cada validador/transformador con sus opciones.
