# 9. View — Plan de construcción

## Objetivo

Construir la capa **View** (headless) que expone un hook para **consumir el resultado** del flujo (transformación, validación y corrección) de forma cómoda para la UI: resultado **paginado**, **filtrado por errores**, **edición** (ya definida en 8. Edit), **exportación** del archivo final y **persistencia opcional** del estado en IndexedDB para recuperar la sesión tras un refresh.

Todo sigue siendo headless: solo datos, hooks y configuración; la UI la construye el consumidor.

---

## Resumen de capacidades del View

| Capacidad | Descripción | Origen |
|-----------|-------------|--------|
| **Resultado paginado** | `getPaginatedResult(page, pageSize)` → `{ page, pageSize, totalCount, rows }` | 8. Edit (useSheetEdit) |
| **API para virtualización** | `getRows(offset, limit)` y `totalRows` — el consumidor puede conectar el hook con **react-window** o **@tanstack/react-virtual** sin renderizar 50k+ filas; escalabilidad sin que el navegador explote | Nuevo en View |
| **Filtrar por errores** | Filtrar filas/hoja por `row.errors.length > 0` o `sheet.errors` para mostrar solo filas con problemas | Nuevo en View |
| **Edición** | `editCell({ rowIndex, cellKey, value })` y pipeline acotada | 8. Edit (useSheetEdit) |
| **Exportar** | Exportar la sheet final (tras transformación, validación y corrección) a CSV/JSON (o formato acordado) | Nuevo en View |
| **Persist (auto-save)** | `persist: boolean` → guardar estado (rawData + correcciones) en **IndexedDB**; al recargar, detectar sesión anterior y permitir "¿Continuar donde lo dejaste?" | Nuevo en View |

## 1. Hook de View: `useSheetView`

El hook agrupa todo lo que la UI necesita para **mostrar** y **actuar** sobre el resultado.

### Contrato propuesto

```ts
interface UseSheetViewOptions {
  /** Initial page (1-based). Optional when using internal pagination state. */
  page?: number;
  /** Default page size for pagination. Hook can manage page/pageSize internally. */
  defaultPageSize?: number;
  /** Filter mode: 'all' | 'errors-only' — show all rows or only rows with errors. */
  filterMode?: 'all' | 'errors-only';
}

interface UseSheetViewReturn<TRow> {
  // —— Resultado y paginación (de useSheetEdit) ——
  /** Sheet actual (validada + transformada + correcciones). */
  sheet: Sheet<TRow> | null;
  /** Resultado paginado según page, pageSize y filterMode. Accepts optional page/pageSize override. */
  getPaginatedResult: (page?: number, pageSize?: number) => PaginatedResult<TRow>;
  /** Filas de la página actual (derived from internal page/pageSize when used). */
  paginatedRows: TRow[];
  /** Current page (1-based). Managed internally when using setPage. */
  page: number;
  /** Set current page. Connects directly to a <Pagination /> component. */
  setPage: (page: number) => void;
  /** Current page size. */
  pageSize: number;

  // —— API para virtualización (escalabilidad) ——
  /** Total de filas en la vista actual (según filterMode: todas o solo con errores). Para react-window / @tanstack/react-virtual. */
  totalRows: number;
  /** Devuelve la rebanada [offset, offset+limit) de la vista actual. Compatible con virtualización: el consumidor no renderiza todas las filas. */
  getRows: (offset: number, limit: number) => TRow[];

  // —— Filtro por errores (memoizado) ——
  /** Filas que tienen al menos un error. Memoized: only recomputed when sheet changes (useMemo). */
  rowsWithErrors: TRow[];
  /** Conteos para la UI: total de filas, filas con error, etc. */
  counts: {
    totalRows: number;
    rowsWithErrors: number;
    totalErrors: number;  // suma de cell/row/sheet errors si se desea
  };

  // —— Edición (de useSheetEdit) ——
  editCell: (params: EditCellParams) => void | Promise<void>;

  // —— Export ——
  /** Exporta la sheet actual a CSV (string o Blob). Para hojas grandes puede ser async (Worker). */
  exportToCSV: (options?: ExportOptions) => string | Blob | Promise<string | Blob>;
  /** Exporta la sheet actual a JSON (string o Blob). Para hojas grandes puede ser async (Worker). */
  exportToJSON: (options?: ExportOptions) => string | Blob | Promise<string | Blob>;
  /** Crea un enlace de descarga para CSV: crea object URL, dispara descarga, revoca URL (revokeObjectURL) para liberar memoria. */
  downloadCSV: (options?: ExportOptions & { filename?: string }) => void | Promise<void>;
  /** Idem para JSON. */
  downloadJSON: (options?: ExportOptions & { filename?: string }) => void | Promise<void>;

  // —— Persist (opcional) ——
  /** Si persist está activo: indica si hay una sesión recuperable en IndexedDB (y no expirada). */
  hasRecoverableSession: boolean;
  /** Recupera el estado guardado y lo carga en el provider (rawData + sheet corregida). */
  recoverSession: () => Promise<void>;
  /** Limpia el estado persistido (por ejemplo tras "exportar y cerrar"). */
  clearPersistedState: () => Promise<void>;
}
```

- **Paginación como estado interno**: el hook puede gestionar `page` y `pageSize` internamente. El consumidor usa `defaultPageSize`, `page`, `setPage`, `paginatedRows` y opcionalmente `getPaginatedResult(page?, pageSize?)` para override. Así `<Pagination />` solo se conecta a `setPage`.
- **rowsWithErrors memoizado**: se calcula con **useMemo** dependiendo solo de `sheet`. Así, al cambiar de página no se vuelve a ejecutar un `.filter()` sobre toda la hoja; el cambio de página es instantáneo incluso con 100.000 filas y filtro por errores.
- **getPaginatedResult**: cuando **filterMode === 'errors-only'**, se aplica sobre `rowsWithErrors` (ya memoizado), no sobre `sheet.rows`.
- **totalRows** y **getRows(offset, limit)**: misma fuente que la paginación (sheet.rows o rowsWithErrors según filterMode). **totalRows** = longitud de esa fuente. **getRows** devuelve `.slice(offset, offset + limit)` sobre ella. Así el consumidor puede usar **react-window** o **@tanstack/react-virtual** sin renderizar 50k+ filas (ver sección 1.5).
- **counts**: derivados para la UI ("50 filas con errores de 1000 total").
- **editCell**: delegado a 8. Edit; solo se reexpone.
- **exportToCSV / exportToJSON**: exportan la sheet actual. Para hojas masivas (>50k filas) la generación puede hacerse en Web Worker para no bloquear la UI (ver sección 2).
- **downloadCSV / downloadJSON**: crean `URL.createObjectURL(blob)`, disparan la descarga y llaman a **URL.revokeObjectURL** después para evitar fugas de memoria.
- **hasRecoverableSession**: si la sesión guardada tiene más de **7 días**, debe ser **false** (no mostrar datos obsoletos). Ver sección 3.
- **recoverSession / clearPersistedState**: solo tienen efecto cuando `persist: true`.

### Implementación sugerida

- **useSheetView** puede **componer** `useSheetEdit` (8. Edit): obtiene `sheet`, `editCell`, `getPaginatedResult` del Edit y añade:
  - **rowsWithErrors** calculado con **useMemo(sheet)** — solo se recalcula cuando la sheet cambia, no en cada render. Así el cambio de página es instantáneo incluso con 100k filas y filtro por errores (evita `.filter()` en cada render).
  - **counts** derivados de sheet y de rowsWithErrors.
  - Estado interno opcional **page** / **pageSize** con `setPage`; **paginatedRows** = resultado de getPaginatedResult(page, pageSize) sobre la fuente correcta (sheet.rows o rowsWithErrors según filterMode).
  - `getPaginatedResult` con filtro: si `filterMode === 'errors-only'`, paginar sobre **rowsWithErrors** (ya memoizado).
  - Funciones de export y download (ver sección 2).
  - Funciones de persist (ver sección 3) si el provider tiene `persist: true`.

## 1.5. Virtualización de datos (escalabilidad)

Si el usuario procesa **50.000 filas**, no puede renderizarlas todas en una tabla de React sin que el navegador explote. La librería es **headless** (no provee la tabla), pero debe exponer una **API compatible con virtualización** para que el desarrollador conecte el hook con **react-window** o **@tanstack/react-virtual** de forma instantánea.

### Contrato

- **totalRows**: número total de filas en la vista actual (según `filterMode`: todas las filas o solo las que tienen errores). La lista virtual usa este valor para el scroll height y el número de ítems.
- **getRows(offset, limit)**: devuelve la rebanada de filas `[offset, offset + limit)` de la misma fuente que la paginación (sheet.rows o rowsWithErrors según filterMode). Es una función **síncrona** y **pura** sobre el array ya memoizado; no recalcula el filtro en cada llamada.

### Uso típico con @tanstack/react-virtual

```ts
const { getRows, totalRows } = useSheetView({ filterMode: 'all' });
const parentRef = useRef<HTMLDivElement>(null);
const rowVirtualizer = useVirtualizer({
  count: totalRows,
  getScrollElement: () => parentRef.current,
  estimateSize: () => 35,
  overscan: 5,
});
const virtualRows = rowVirtualizer.getVirtualItems();
const rows = getRows(virtualRows[0]?.start ?? 0, (virtualRows[virtualRows.length - 1]?.end ?? 0) - (virtualRows[0]?.start ?? 0));
// Renderizar solo `rows` en lugar de 50.000 elementos.
```

### Criterios de aceptación (virtualización)

- [x] **totalRows** refleja el tamaño de la vista actual (todas las filas o solo con errores según filterMode).
- [x] **getRows(offset, limit)** devuelve exactamente la rebanada correspondiente; no modifica estado; compatible con llamadas frecuentes desde un virtualizer.
- [x] Documentación o tipos que indiquen compatibilidad con react-window y @tanstack/react-virtual.

## 2. Exportación del archivo final

Tras el proceso de transformación, validación y corrección manual, el usuario debe poder **exportar** el resultado.

### Contrato de exportación

```ts
interface ExportOptions {
  /** Incluir fila de cabecera (nombres de campo). Default true. */
  includeHeaders?: boolean;
  /** Separador para CSV. Default ','. */
  csvSeparator?: string;
  /** Nombre de archivo sugerido (sin extensión). */
  filename?: string;
  /** Si true, valores de fecha se exportan en formato local (human-friendly). Default false. */
  formatDatesForExport?: boolean;
}

// Uso desde el hook
exportToCSV(options?: ExportOptions): string | Blob | Promise<string | Blob>;
exportToJSON(options?: ExportOptions): string | Blob | Promise<string | Blob>;
downloadCSV(options?: ExportOptions & { filename?: string }): void | Promise<void>;
downloadJSON(options?: ExportOptions & { filename?: string }): void | Promise<void>;
```

- **CSV**:
  - Recorrer `sheet.rows` según `sheetLayout` (orden de fields); primera línea con headers si `includeHeaders`.
  - Usar `csvSeparator`; **escapar comillas** si el valor contiene el separador o comillas (RFC 4180-style: envolver en comillas y duplicar comillas internas).
  - **BOM para Excel**: incluir **Byte Order Mark** (`\uFEFF`) al inicio del string o del Blob para que Excel reconozca UTF-8 y caracteres como ñ/tildes se vean bien al abrir el CSV.
  - **Transformaciones de formato**: si `formatDatesForExport` (Fase B), aplicar formato local a fechas para que el archivo exportado sea amigable para humanos.
- **JSON**: serializar la sheet o solo `rows`; opción "solo datos" vs "sheet completa".
- **downloadCSV / downloadJSON**: crear `URL.createObjectURL(blob)`, disparar descarga (anchor click), luego **URL.revokeObjectURL** para liberar memoria.

### Dónde vive la lógica

- **`core/view/export/`**: funciones puras `sheetToCSV(sheet, sheetLayout, options)` y `sheetToJSON(sheet, options)` (BOM, escape, formatDatesForExport). Para hojas >50k filas, un Worker puede invocar la misma lógica y devolver Blob/string al Main Thread.
- El hook **useSheetView** llama a estas funciones (o al Worker) con la **sheet actual** del provider y expone **downloadCSV**/**downloadJSON** con revokeObjectURL.

### Exportación asíncrona para archivos gigantes

- **Problema**: Con hojas masivas (>50.000 filas), generar el string CSV/JSON en el Main Thread puede tardar 1–2 segundos y bloquear la UI (dropped frames).
- **Mejora**: Para hojas por encima de un umbral (p. ej. 50.000 filas), reutilizar un **Web Worker** para generar el string CSV/JSON. El Main Thread solo solicita la exportación y recibe el Blob listo para descargar; cero bloqueo de UI.
- **Contrato**: `exportToCSV` / `exportToJSON` pueden devolver `Promise<string | Blob>` cuando se use el Worker; el consumidor puede usar `downloadCSV()` que internamente espera el Blob y luego hace createObjectURL + revokeObjectURL.

### Criterios de aceptación (export)

- [x] Exportar la sheet actual (validada + transformada + correcciones) a CSV con opciones (headers, separador, BOM, escape de comillas).
- [x] CSV con **BOM** (`\uFEFF`) para correcta apertura en Excel (UTF-8).
- [x] sheetToCSV capaz de transformaciones de formato final (p. ej. fechas en formato local) cuando `formatDatesForExport`.
- [x] Exportar la sheet actual a JSON (formato acordado: solo datos o sheet completa).
- [x] **downloadCSV()** (y downloadJSON): crear object URL, disparar descarga, **revokeObjectURL** para no dejar fugas de memoria.
- [ ] Para hojas grandes (>50k filas), exportación en Worker para no bloquear la UI (opcional / fase posterior).
- [x] No modificar el estado del provider; solo lectura de la sheet y generación de string/Blob.

## 3. Persistent State (Auto-save) con IndexedDB

Objetivo: si el usuario sube un archivo, corrige 50 errores y **refresca la página**, poder preguntarle **"¿Quieres continuar donde lo dejaste?"** y restaurar rawData + sheet con correcciones.

### Por qué IndexedDB y no LocalStorage

- **LocalStorage** tiene límite bajo (~5–10 MB) y es síncrono; un sheet grande (muchas filas/celdas) puede exceder el tamaño.
- **IndexedDB** permite almacenar más datos y es asíncrono; adecuado para estado grande (rawData + sheet).

### Configuración en el Provider (Setting)

En la configuración del provider (2. Setting / ImporterProvider) se añade una opción:

```ts
interface ImporterProviderProps {
  // ... existentes (sheetLayout, children, etc.)
  /**
   * If true, the current state (rawData + result sheet with corrections) is persisted to IndexedDB.
   * On mount, if a previous session is found, the consumer can prompt "Continue where you left off?" and call recoverSession().
   */
  persist?: boolean;
  /**
   * Optional. Key used for IndexedDB store (e.g. per-upload or per-user).
   * If not provided, a default key (e.g. 'react-import-sheet-session') is used.
   */
  persistKey?: string;
}
```

- **persist**: si `true`, después de cada cambio relevante (por ejemplo: resultado listo, o tras cada `editCell`) se guarda en IndexedDB el estado necesario para recuperar.
- **persistKey**: permite varias "sesiones" (por ejemplo por usuario o por tipo de importación). Si no se provee, se usa una clave por defecto.

### Qué persistir

Para poder "continuar donde lo dejaste", hay que guardar:

- **rawData** (o el resultado del Parser): lo que se subió originalmente (RawSheet o equivalente), para no depender del File en disco.
- **sheet actual**: el resultado validado + transformado + correcciones (el mismo objeto sheet que tiene el provider). Así al recuperar se restaura exactamente el estado de la tabla.

Opcional: **status**, **fileName**, **sheetLayout** (o su referencia) si se quieren restaurar también la barra de estado o el nombre del archivo. Para un MVP basta **rawData + sheet**.

### Flujo

1. **Al montar el provider** (con `persist: true`):
   - Consultar IndexedDB con `persistKey` (o default).
   - Si hay datos guardados **y** el **timestamp** de la sesión no ha expirado (p. ej. **≤ 7 días**) → no restaurar automáticamente; marcar **hasRecoverableSession = true** para que la UI decida.
   - Si la sesión tiene **más de 7 días**, considerar **hasRecoverableSession = false** (no mostrar datos obsoletos); opcionalmente borrar esa entrada al detectarla.
   - La UI muestra por ejemplo: "Tienes una sesión anterior. ¿Continuar donde lo dejaste?" con [Sí] [No].

2. **Usuario pulsa "Sí"**:
   - Se llama **recoverSession()** desde el hook (useSheetView o un usePersist dedicado).
   - Se lee el estado desde IndexedDB y se escribe en el provider: rawData + sheet (y opcionalmente status/fileName).
   - hasRecoverableSession pasa a false (o se limpia la entrada en IndexedDB tras recuperar).

3. **Usuario pulsa "No"**:
   - Se llama **clearPersistedState()** para borrar la sesión guardada y seguir con flujo normal (nueva subida).

4. **Mientras trabaja** (con persist: true):
   - **Save buffer (debounce)**: no guardar en cada **editCell**. Usar un **debounce de 2–3 segundos**: tras la última edición (o tras parser/transform listo), se escribe una sola vez en IndexedDB. Así se evitan cientos de escrituras consecutivas (megabytes) que podrían bloquear la UI.
   - Opcional: guardar una versión **comprimida** o solo **rawData + delta** de las ediciones para reducir tamaño y acelerar la carga; para MVP basta rawData + sheet con debounce.

5. **Al exportar y "cerrar"** (opcional):
   - El consumidor puede llamar **clearPersistedState()** después de exportar para que la próxima visita no vea "¿Continuar?".

### Contrato del módulo de persistencia

```ts
// core/view/persist/ o utils/persist/
const PERSIST_STORE_NAME = 'react-import-sheet-persist';

const PERSIST_SESSION_MAX_AGE_MS = 7 * 24 * 60 * 60 * 1000; // 7 days

interface PersistedState {
  rawData: RawSheet;  // o el tipo que use el provider
  sheet: Sheet;       // sheet actual (validada + transformada + correcciones)
  savedAt: number;    // timestamp (Date.now()) — usado para expiración
  fileName?: string;  // opcional
  persistKey?: string;
}

// hasRecoverableSession debe ser false si loadPersistedState retorna null o si savedAt < (now - 7 days).

async function savePersistedState(key: string, state: PersistedState): Promise<void>;
async function loadPersistedState(key: string): Promise<PersistedState | null>;
async function clearPersistedState(key: string): Promise<void>;
```

- **key**: el `persistKey` del provider o el default.
- **savePersistedState**: serializar y guardar en IndexedDB (rawData + sheet pueden ser objetos grandes; IndexedDB acepta structured clone).
- **loadPersistedState**: devolver null si no hay nada; si hay datos, el hook expone hasRecoverableSession = true y recoverSession() los carga en el provider.
- **clearPersistedState**: borrar la entrada para ese key.

### Hook / contexto para persist

- Opción A: **useSheetView** incluye `hasRecoverableSession`, `recoverSession`, `clearPersistedState` cuando `persist === true`.
- Opción B: **usePersistSession()** separado que solo expone persist; y useSheetView lo usa internamente o el consumidor usa ambos hooks.

Recomendación: **incluir en useSheetView** para que la UI tenga un solo hook para "vista + export + persist". Si el provider no tiene `persist: true`, esas funciones pueden ser no-op y hasRecoverableSession siempre false.

### Criterios de aceptación (persist)

- [x] Con `persist: true`, el estado (rawData + sheet) se guarda en **IndexedDB** (no LocalStorage).
- [x] Clave configurable con `persistKey` (o valor por defecto).
- [x] **Debounce de 2–3 s**: no guardar en cada editCell; una sola escritura tras el debounce (y tras parser/transform listo).
- [x] **Expiración**: si la sesión guardada tiene más de **7 días** (`savedAt`), **hasRecoverableSession** debe ser **false** (no mostrar datos obsoletos).
- [x] Al montar, si existe sesión guardada y no expirada, **no** se restaura automáticamente; se expone **hasRecoverableSession** para que la UI pregunte.
- [x] **recoverSession()** restaura rawData + sheet en el provider.
- [x] **clearPersistedState()** borra la sesión guardada.
- [x] Tests unitarios (Vitest) para save/load/clear y lógica de expiración con IndexedDB (o mock).

## 7. Criterio de éxito técnico

- **Zero UI blocking**: Ni la exportación de 100k filas ni el guardado en IndexedDB deben causar un "dropped frame" (bajón de FPS) en la interfaz. Exportación masiva en Worker; persist con debounce.
- **Robustez**: Si el usuario corta la luz mientras edita, al volver su progreso debe estar ahí (persist + recoverSession).

Con esta capa View, la librería queda lista para uso en el mundo real.
