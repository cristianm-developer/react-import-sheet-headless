# 7. Transform — Plan de construcción (Revisado)

## Objetivo

Construir el módulo de **transformación** que tome la sheet validada y aplique cambios de valor **únicamente** en las celdas, filas y hojas que no contengan errores. Para maximizar la eficiencia, las transformaciones de **Celda** y **Fila** serán **síncronas** y se procesarán por **lotes (chunks)**; la asincronía queda exclusivamente para la transformación de **Hoja**. El proceso corre en un **Web Worker**; el Worker devuelve un **Delta de valores** (no la sheet completa) y el Main Thread aplica esos cambios sobre su copia local, reduciendo uso de memoria y tiempo de transferencia.

---

## Entrada y salida

| Concepto | Origen | Descripción |
|----------|--------|-------------|
| **Entrada** | Validator (6. Validator) + Setting (2. Setting) | **sheet** (output del validador: `name`, `sheetLayout`, `filesize`, `mimetype`, `extension`, `errors`, `rows` con `row` → `{ index, errors, cells }` y **cell** → `{ key, value, errors }`) y **sheetLayout** (plantilla con `fields` y en cada field **identificadores** de transformaciones, p. ej. `transformations: ['toUpperCase', 'formatCurrency']`, más `row-transformations` y `sheet-transformations` como identificadores). |
| **Salida (Worker)** | Tipos del Setting | **Delta de valores**: `deltas: Array<{ row: number, col: string, newValue: unknown }>` (u equivalente por índice de columna). El Worker **no** devuelve un segundo objeto sheet completo. |
| **Salida (Main Thread)** | — | El **Delta Applier** recibe los deltas y actualiza la sheet en el Provider (solo las filas/celdas afectadas; inmutabilidad solo donde toque para React). |

El transform **no valida** ni cambia errores; solo aplica transformaciones de valor donde está permitido (sin error en ese nivel).

## Ajustes de rendimiento y arquitectura

### 1. Sincronía y chunks (alineado con el Validator)

- **Regla:** **Cell and Row transforms MUST be synchronous** for performance. Only **table-level** (sheet) transforms may be **async** (e.g. sending the result to a backend for pre-processing or enrichment). Same rationale as Validator: predictable performance in the loop; batching at table level (one request with full data instead of thousands). Ver **Architecture.md** — *Sync vs async* y **6. Validator** — *Eliminación del overhead de promesas*.
- **Estrategia:** Se procesan las filas en **chunks** (lotes de 100–500). Cell y row transforms **síncronos** en el bucle; **sheet-transforms** (table) se ejecutan **una sola vez al final** y pueden ser async. Si un table-transform llama al backend, debe ser **resiliente** (try/catch, código de error tipo **`EXTERNAL_TRANSFORM_FAILED`**, **AbortController** para cancelar el fetch). La respuesta del backend debe seguir el **mismo esquema** que el resto: errores (SheetError[]) o resultado actualizado para mostrar en la sheet.

### 2. Registry pattern para transforms

- Igual que con los validadores: el **sheetLayout no envía funciones** al Worker.
- El Worker **importa** un **TransformRegistry** (p. ej. `TransformRegistry.ts`).
- El layout envía **identificadores**: `transformations: ['toUpperCase', 'formatCurrency']`.
- Así el procesamiento ocurre íntegramente en el Worker sin costosos proxies de Comlink por cada celda.

**Patrón Register() (built-in y custom):** Cada transform built-in (cell, row, sheet) expone **`Register(registerTransform?)`** que devuelve el identificador string y opcionalmente registra la implementación en el Provider/Worker si se pasa el callback. El layout puede usar `transformations: [CellToUpperCaseTransform.Register(registerTransform), 'formatCurrency']`. La librería expone interfaces **`CellTransformDefinition`**, **`RowTransformDefinition`**, **`SheetTransformDefinition`** para que el usuario cree sus propios transforms con el mismo contrato. Ver **Architecture.md** — *Register() pattern* y **2. Setting** — *Patrón Register()*.

**Tipo en el Registry:** Cada entrada del Registry de transforms debe llevar **`type: 'cell' | 'row' | 'table'`** para que el Runner invoque **cell/row** en el bucle por filas y **table** una sola vez al final (con acceso al dataset completo). Ver **Architecture.md** — *Registry entry type*.

**Funciones puras y params:** Si se usa serialización al Worker (`fn.toString()`), los transforms deben ser puros. Toda configuración (formatos, opciones) debe pasarse por **params** en el layout. Ver **Architecture.md** — *Technical note: Serialization to Worker and pure functions*.

### 3. Estrategia Delta (eficiencia de datos)

- **Cambio crítico**: El Worker **no devuelve** un segundo objeto sheet completo.
- **Solución**: El Worker devuelve un **Delta de valores**: `deltas: [{ row, col, newValue }, ...]`.
- El **Main Thread** aplica estos cambios sobre su copia local de la sheet (utilidad **Delta Applier**). Se reduce drásticamente el uso de memoria y el tiempo de transferencia.

## Reglas de aplicación (capa de protección — Safe-first)

El motor de transformación debe ser **"miedoso"** (Safe-first):

- **Sheet**: Si `sheet.errors.length > 0`, se **cancela todo** el proceso de transformación (no se ejecuta ninguna transformación).
- **Fila**: Si una fila tiene errores, se **salta por completo** (optimización de ciclos de CPU).
- **Celda**: Si una celda tiene errores, **no** se le aplica ningún transformador.

**Atomicidad con validaciones de nivel table:** El proceso de Transform **solo debe iniciar** si las **validaciones de nivel table** (especialmente las de integridad) han pasado, **o** si el usuario acepta explícitamente continuar con errores. Es decir: (1) el Validator ejecuta cell → row → **table**; (2) si las table-validations reportan errores (ej. duplicados, integridad), el flujo puede **no** ejecutar Transform hasta que se corrijan los datos o el usuario elija "Continuar con errores"; (3) solo entonces se ejecuta Transform (que ya aplica Safe-first por sheet/row/cell). Así se evita transformar datos que fallan integridad a nivel hoja. Ver **6. Validator** — *Validaciones de nivel table: atomicidad* y **Architecture.md** — *Table-level validations*.

El resultado (deltas aplicados o ninguno si hubo sheet.errors) se coloca en el **provider** vía Delta Applier y se actualiza el **status**. El **progreso** se emite vía EventTarget.

---

## Orden de ejecución optimizado (Worker)

1. **Validación inicial**: Si la hoja tiene errores globales (`sheet.errors.length > 0`), salir inmediatamente; retornar `deltas: []` (o señal equivalente).

2. **Bucle por chunks (síncrono)**:
   - Para cada **lote de filas** (chunk de 100–500):
     - Si la fila tiene `row.errors.length > 0`, **saltar** esa fila.
     - Ejecutar **cell-transforms** (sync) solo en celdas sin error.
     - Ejecutar **row-transforms** (sync) para esa fila (si no tenía row errors).
     - Acumular los cambios en el objeto **deltas** (coordenadas + nuevo valor).
   - Emitir **progreso** vía `onProgress` con **throttling** (máximo cada ~16 ms).

3. **Finalización (async, opcional)**:
   - Ejecutar **sheet-transforms** (table) sobre el estado ya transformado (si el layout lo define). **Solo estos** pueden ser async (backend). En el runner: **try/catch**; en fallo devolver error con código **`EXTERNAL_TRANSFORM_FAILED`**; pasar **AbortController** para cancelar fetch en **`abort()`**. La respuesta (errores o resultado actualizado) debe usar el mismo esquema que el resto (deltas / SheetError).
   - **Retorno**: Enviar el array de **deltas** al Main Thread (no la sheet completa).

---

## Web Worker y no bloquear al usuario (Comlink)

- Toda la lógica de transformación (chunks, cell/row transforms sync, acumulación de deltas) corre dentro de un **Web Worker**. La comunicación se hace con **Comlink**. La capa que invoca al Worker debe aplicar **timeout y recuperación** (ver **Architecture.md** — *Maturity & resilience → Worker timeout & recovery*).
- **Abort y ciclo de vida:** El hook que instancia el Worker (p. ej. `useTransformWorker`) debe guardar la instancia en una **ref** para **`abort()`** y en el **cleanup de `useEffect`** al desmontar llamar **`worker.terminate()`**. Ver **Architecture.md** — *Process Abort & Worker Cleanup* y **2. Setting** — *Abort y ciclo de vida*.
- En el worker: **`Comlink.expose(api)`** con un método (p. ej. `transform(sheet, sheetLayout, options?, onProgress?)`) que ejecute el orquestador y devuelva **`Promise<{ deltas }>`** (no `Promise<sheet>`). El progreso se reporta con **`Comlink.proxy(onProgress)`** y **throttling** (máx. cada ~16 ms).
- En el main thread: **`Comlink.wrap(worker)`** para obtener el proxy; `await transformProxy.transform(sheet, sheetLayout, options, onProgress)`; al resolver, recibir **deltas** y pasarlos al **Delta Applier** para actualizar la sheet en el provider.
- El hilo principal no se bloquea. **Progreso**: no se guarda en el Context; el hook **dispara un evento** en el **EventTarget** (véase siguiente sección).

---

## Delta Applier (Main Thread)

- **Utilidad**: Una función o módulo que reciba el array de **deltas** y la **sheet actual** en el Provider, y actualice solo las celdas indicadas.
- **Inmutabilidad**: Aplicar cambios de forma que React detecte actualizaciones solo en las filas/estructuras afectadas (evitar clonar la sheet entera si no es necesario).
- **Ubicación sugerida**: p. ej. `transform/delta-applier.ts` o dentro del hook que orquesta Worker + Provider.

---

## Estado y actualización en el provider

- El resultado del transform se aplica en el **provider** mediante el **Delta Applier**: se parte de la sheet validada y se aplican los **deltas** devueltos por el Worker (p. ej. campo `validatedAndTransformedSheet` o equivalente).
- El **status** del proceso se actualiza (`'transforming'` durante el trabajo, `'success'` o `'error'` al terminar).
- El **progreso** se emite **solo** vía **EventTarget**; no se guarda en Context para no provocar re-renders en el Provider.

---

## Barras de progreso (EventTarget)

Se mantiene el **desacoplamiento total** de React:

- **Evento**: `importer-progress`.
- **Payload**: `{ phase: 'transforming', globalPercent, localPercent, currentRow, totalRows }`.
- **UI**: Solo el componente de la barra de progreso se suscribe y actualiza mediante una ref o estado local mínimo (sin que el Provider re-renderice por progreso).

---

## Alcance funcional (resumen)

- **Entrada**: sheet (output del validador) + sheetLayout (identificadores de transforms: cell, row, sheet).
- **Condiciones (Safe-first)**: sheet.errors → cancelar todo; row.errors → saltar fila; cell.errors → no transform en esa celda.
- **Ejecución**: en Web Worker; **cell** y **row** transforms **síncronos** por **chunks**; solo **sheet-transforms** puede ser async. Orquestador con "saltar si hay error" y acumulación de **deltas**.
- **Salida**: **deltas** (array de coordenadas + nuevo valor); el Main Thread aplica con **Delta Applier** sobre la copia local en el provider.
- **Estado**: sheet actualizada en el provider vía deltas; status actualizado. **Progreso**: vía EventTarget (`importer-progress`); sin re-renders en el Provider.
- **Tests**: tests unitarios para Registry, Runner, Delta Applier y reglas de error (cobertura ≥ 80 % en casos de error).

## Arquitectura del transform

### 1. TransformRegistry (Worker)

- **TransformRegistry.ts**: registro de funciones de transformación **síncronas** (limpieza, formateo, casting final) por identificador. El Worker importa este módulo; el layout solo envía nombres: `transformations: ['toUpperCase', 'formatCurrency']`.
- Las implementaciones reutilizables viven en **`src/utils/controller/[context]/`** (p. ej. `cell-email-transform.ts`). El Registry las agrega por nombre para que el Worker las resuelva sin recibir funciones por Comlink.

### 2. Orquestador (Runner) dentro del worker

- **runCellTransforms(cell, field, sheetLayout)** — **sync**  
  Si `cell.errors.length === 0`, ejecuta en secuencia los transforms del field (resueltos desde el Registry por identificador); cada uno recibe valor/celda y devuelve el nuevo valor. No async.

- **runRowTransforms(row, sheetLayout)** — **sync**  
  Si `row.errors.length === 0`, ejecuta en secuencia los row-transforms del layout (desde Registry). Recibe fila, devuelve fila actualizada.

- **runSheetTransforms(sheet, sheetLayout)** — **async (opcional)**  
  Si `sheet.errors.length === 0`, ejecuta en secuencia los sheet-transforms (p. ej. envío a API). Único punto donde se permite async.

- **runTransform(sheet, sheetLayout, onProgress?)**  
  Si hay sheet.errors, retorna `{ deltas: [] }` de inmediato. Si no: bucle por **chunks** de filas → por cada fila sin row.errors: runCellTransforms (solo celdas sin error), runRowTransforms; **acumular deltas** (row, col, newValue). Throttle de onProgress (~16 ms). Al final, opcionalmente runSheetTransforms. Retorno: **`{ deltas }`**, no la sheet completa.

### 3. Web Worker (Comlink)

- **API**: `transform(sheet, sheetLayout, options?, onProgress?)` → **`Promise<{ deltas }>`**. Progreso vía **`Comlink.proxy(onProgress)`** con throttling. El Worker **no** recibe funciones; usa **TransformRegistry** con identificadores del layout.

### 4. Delta Applier (Main Thread)

- Recibe **deltas** y la **sheet** actual; actualiza solo las celdas indicadas. Inmutabilidad solo en las filas/ramas afectadas para que React actualice de forma eficiente.

### 5. Hook / capa principal

- **useTransformWorker** (o equivalente):
  - Instancia el worker con **`Comlink.wrap(worker)`**.
  - Expone `transform(sheet, sheetLayout, options?)` que llama al proxy y recibe **`{ deltas }`**; invoca al **Delta Applier** para actualizar la sheet en el provider.
  - **Status**: idle | transforming | success | error. **Progreso**: al recibir el callback del worker, **disparar evento** `importer-progress` en el EventTarget (no guardar en Context).

## Plan de tareas (orden sugerido)

### Registry

- **TransformRegistry.ts**: Crear registro con funciones **síncronas** de transformación (limpieza, formateo, casting final). El layout envía identificadores; el Worker importa el Registry y resuelve por nombre.

### Runner (orquestador)

- Implementar el orquestador que maneje la lógica **"Saltar si hay error"** (sheet → salir; row → saltar fila; cell → no transform) y el **procesamiento por lotes (chunks)** de 100–500 filas. Cell y row transforms sync; acumulación de **deltas**; onProgress con throttling (~16 ms). Retorno: `{ deltas }`.

### Delta Applier

- Crear utilidad en el **Main Thread** que reciba los **deltas** y actualice la sheet en el Provider de forma eficiente (inmutabilidad solo en las filas afectadas para React).

### Tipos y contrato

- Tipos para **sheet**, **row**, **cell**, **sheetLayout** (transformations como array de identificadores). Tipos para **deltas** (`{ row, col, newValue }`) y mensajes del worker (`TransformProgress`, `TransformSuccess`, `TransformError`).

### Web Worker (Comlink)

- **transform.worker.ts**: expone `transform(sheet, sheetLayout, options?, onProgress?)` → `Promise<{ deltas }>`. Usa TransformRegistry; no recibe funciones. Configurar entry en el bundler.

### Hook y provider

- **useTransformWorker**: instancia worker, proxy Comlink; al resolver con `{ deltas }`, invocar **Delta Applier** y actualizar provider. Progreso: disparar evento `importer-progress` en EventTarget (sin guardar en Context). Documentar contrato del evento.

### Tests de regla y cobertura

- **Vitest**: Verificar que si una **celda tiene un error**, su valor **no cambia** aunque tenga transformadores asignados. Igual para fila y hoja. Cobertura de tests para los transformadores **al menos 80 %**, con especial atención a **casos de error** (sheet/row/cell con errors).
- Tests del runner (chunks, deltas, orden), Delta Applier (aplicación correcta de deltas) y, si aplica, worker/hook.

## Tests unitarios (contexto Transform)

Se deben crear **tests unitarios con Vitest** para el módulo Transform. Alcance:

- **Regla crítica**: Si una **celda** tiene error, su **valor no cambia** aunque tenga transformadores asignados. Igual para fila (no se aplican transforms a esa fila) y hoja (no se ejecuta ningún transform).

- **Cobertura**: al menos **80 %** para los transformadores, con **énfasis en casos de error** (cell/row/sheet con errors).

**Ubicación**: tests junto al código (p. ej. `*.test.ts` en `transform/runner/`, `transform/delta-applier`, worker, hooks) o en `transform/__tests__/`. **Criterio de éxito**: suite Vitest en verde y cobertura ≥ 80 % en casos de error antes de cerrar el paso.

## Criterios de aceptación (resumen)

- [ ] Las transformaciones de **Celda** y **Fila** son **estrictamente síncronas** (MUST be synchronous). Solo los **table** (sheet) transforms pueden ser async (mismo esquema Register() y resiliencia que table validators).
- [ ] Table transforms async: **try/catch** en el runner; en fallo de backend **SheetError** con **`EXTERNAL_TRANSFORM_FAILED`**; **AbortController** para cancelar fetch en **`abort()`**; respuesta del backend con el mismo esquema (errores o resultado actualizado).
- [ ] El Worker devuelve un **Delta** de cambios (`deltas`), **no** la hoja completa.
- [ ] Si la hoja / fila / celda tiene errores, las transformaciones correspondientes **no se ejecutan** (Safe-first).
- [ ] El progreso se emite vía **EventTarget** (`importer-progress`) **sin provocar re-renders** en el Provider.
- [ ] La cobertura de tests para los transformadores es de **al menos el 80 %**, con especial atención a **casos de error** (celda/fila/hoja con errors).
- [ ] Entrada: sheet (output del validador) + sheetLayout con **identificadores** de transformations (Registry en Worker).
- [ ] Transform ejecutado en Web Worker vía **Comlink**; resultado aplicado en el provider mediante **Delta Applier**.
- [ ] **Patrón Register():** Cada transform built-in (cell, row, sheet) exporta `id`, `type: 'cell'|'row'|'table'` y `Register(registerTransform?)`; al registrar se guarda **type** en el Registry para que el Runner invoque cell/row por fila y table una vez al final. Interfaces **CellTransformDefinition**, **RowTransformDefinition**, **SheetTransformDefinition** expuestas para transforms custom (ver Architecture y 2. Setting).
- [ ] **Atomicidad:** Transform solo se ejecuta si las validaciones de nivel table han pasado o el usuario acepta continuar con errores (ver 6. Validator — table-level).
- [ ] **Abort y cleanup:** El hook guarda el Worker en ref; cleanup en desmontaje llama `worker.terminate()`; el Provider puede terminar este Worker vía `abort()`.
