# 6. Validator — Plan de construcción (Revisado)

## Objetivo

Ejecutar la lógica de **validación** sobre el **RawSheet** (ya sanitizado) mediante un **Web Worker**. Se prioriza la ejecución **síncrona por lotes** para maximizar la velocidad, utilizando un **sistema de registro (Registry)** para evitar el costo de serialización de funciones y una estrategia de comunicación **Delta** para proteger la memoria del hilo principal.

---

## Ajustes estratégicos (puntos críticos)

### 1. Serialización y Registry Pattern

**Problema:** Las funciones no pueden viajar al Worker vía `postMessage`. Usar proxies de Comlink para funciones individuales destruiría el rendimiento (miles de llamadas de ida y vuelta al Main Thread).

**Solución:** El `sheetLayout` solo contendrá **identificadores (strings)** (ej: `validators: ['required', 'is-email']`). El Worker importará un **ValidatorRegistry** que mapea estos strings a funciones reales. El procesamiento ocurre **100% dentro del Worker**.

### 2. Eliminación del overhead de promesas (regla sync/async)

**Problema:** Ejecutar 50.000 celdas asíncronamente satura la cola de microtareas (microtask queue). Además, hacer una petición por fila al backend (10.000 filas → 10.000 peticiones) es ineficiente y frágil.

**Solución:** **Cell and Row validators MUST be synchronous** for performance and predictability; the bulk of the work (thousands of rows) stays linear and does not depend on network or event loop. The **only** validators allowed to be **async** are **table-level** (sheet) validators, which run **once at the end** with the full dataset — so any backend call is **batched** (e.g. one request with 10,000 IDs instead of 10,000 requests). Ver **Architecture.md** — *Sync vs async* y *Data flow*.

**Regla para Cursor/implementación:** En **Validator.md** y **Architecture.md** debe quedar explícito: *"Cell and Row validators MUST be synchronous for performance. Only Table validators are allowed to be async for backend/database integrity checks."*

### 3. Estrategia Delta por defecto

**Problema:** Devolver una hoja completa de 20MB al Main Thread causa un bloqueo (freeze) durante la deserialización del mensaje.

**Solución:** El Worker **no devuelve la hoja**. Devuelve únicamente un **array de errores** con coordenadas y metadatos:

```ts
errors: [{ row: 10, col: 'email', code: 'INVALID', params: { value: '...' } }]
```

El Main Thread **parchea** estos errores sobre la copia de los datos que ya posee.

## Entrada y salida

| Concepto | Origen | Descripción |
|----------|--------|-------------|
| **Entrada** | Parser + Sanitizer + Setting | `RawSheet` (ya sanitizado; tipos listos para validación) y `sheetLayout` con **identificadores** de validadores por nivel (ej. `validators: ['required', 'is-email']`). |
| **Salida** | Delta de errores | **No** se devuelve la sheet. Se devuelve un **delta de errores**: array de errores con coordenadas (`row`, `col` o nivel) y `SheetError`. El Main Thread aplica este delta sobre su copia local de la sheet. |

El parser/sanitizer entrega datos con layout reconocido y tipos ya castiados; el validador en el Worker itera por chunks, ejecuta validadores síncronos desde el Registry y acumula solo los errores para enviarlos como delta.

## Arquitectura del validador

### 1. Estructura de errores (I18n: código + params)

**Nunca** devolver mensajes de error como strings estáticos (ej. "Email inválido"): acoplaría la librería a un idioma. Ver **Architecture.md** — *Maturity & resilience → I18n of errors*.

El contrato de error es **siempre** un objeto con código y metadatos; la UI traduce el `code` (y opcionalmente `params`) al idioma del usuario:

```ts
interface SheetError {
  code: string;           // Ej: 'INVALID_EMAIL', 'MIN_LENGTH'
  level: 'error' | 'fatal' | 'warning';
  params?: Record<string, unknown>; // Ej: { value, min, actual }
  message?: string;       // Fallback opcional (depuración); no usar para UI localizada
}
```

Los validadores deben devolver **solo** `SheetError[]` con `code` y `params`; la UI decide el texto final.

### 2. Cortafuegos (Early Exit)

Si un validador devuelve un nivel **fatal** o falla una validación marcada como **crítica** en el layout, el orquestador aplica **Early Exit** para esa fila o columna, abortando ejecuciones posteriores para ahorrar ciclos de CPU.

### 3. Orquestador de validación (Worker)

El Runner usa el **tipo** de cada entrada del Registry (`type: 'cell' | 'row' | 'table'`) para decidir cuándo invocar: **cell** y **row** por cada fila/celda; **table** una sola vez al final, sin tener que programar la lógica manualmente para cada importación. Ver **Architecture.md** — *Registry entry type*.

Orden estricto del pipeline:

1. **Fase síncrona (cómputo):** Recorre todas las filas ejecutando solo validadores **cell** y **row** (síncronos). Genera el primer mapa de errores (delta).
2. **Fase asíncrona (integridad):** Si (y solo si) el usuario definió validadores con **`type: 'table'`**, el Worker ejecuta **`await`** de esas funciones, con acceso al dataset completo. Es el **único** punto asíncrono.
3. **Consolidación:** Se combinan los errores locales (cell/row) con los errores devueltos por los validadores de table (incl. backend).

Detalle por fase:

- **Fase de chunks (Sync):**
  - Itera filas en bloques (ej. cada 100 filas).
  - Para cada celda/fila: ejecuta solo validadores registrados con **`type: 'cell'`** o **`type: 'row'`** (resolviendo por nombre desde el layout). Todos son **síncronos**.
  - Reporta progreso al Main Thread mediante `onProgress`.

- **Fase final (table, una sola vez):**
  - Ejecuta **validadores con `type: 'table'`** (sheet-validators) **una sola vez** al terminar el bucle, con acceso al **dataset completo** ya procesado en el Worker. **Solo estos** pueden ser **async** (ej. integridad contra API externa). Ver *Manejo de errores de red* más abajo.

- **Fase de retorno:**
  - Envía **únicamente el delta de errores** (coordenadas + `SheetError`), incluyendo errores de nivel table o multi-fila (ver *Validaciones de nivel table* más abajo).

### 4. Validaciones de nivel table: scope, output y atomicidad

Las validaciones registradas con **`type: 'table'`** (sheet-validators) tienen un contrato distinto al de cell/row:

- **Scope:** Se ejecutan **una sola vez** al final del ciclo en el Worker, con acceso al **dataset completo** ya procesado (todas las filas y celdas). Pueden inspeccionar la hoja entera para reglas de integridad (unicidad, referencias cruzadas, totales).
- **Output:** Pueden generar **errores que afectan a múltiples filas** (ej. "filas 5 y 12 duplican el mismo ID") o **errores globales del Sheet** (ej. `sheet.errors` o un error que no se asocia a una celda concreta). El delta de errores y la estructura de la sheet deben soportar errores a nivel hoja y errores multi-fila.
- **Atomicidad:** El proceso de **Transform** (paso 7) solo debería iniciar si las validaciones de nivel **table** (en especial las de integridad) han pasado, **o** si el usuario acepta explícitamente continuar con errores. Es decir: (1) el Runner ejecuta cell → row → **table**; (2) si hay errores de table (p. ej. duplicados), el sistema puede **bloquear** el paso a Transform hasta que se corrijan los datos, o ofrecer al usuario la opción "Continuar con errores"; (3) solo entonces se ejecuta Transform (que ya salta filas/celdas con error). Así se evita transformar datos que fallan validaciones de integridad a nivel hoja. Ver **Architecture.md** — *Table-level validations* y **7. Transform** — reglas Safe-first y atomicidad.

### 5. Manejo de errores de red (validadores table async)

Al ser el **único** punto asíncrono, el Worker debe protegerse de quedar colgado si el backend no responde. En el módulo que ejecuta las validaciones de nivel table (p. ej. **`run-table-validation.ts`** o equivalente):

- **Try/catch obligatorio:** Si el `fetch` (o la llamada al backend) falla — error 500, sin conexión, timeout — la función debe **capturar el error** y devolver un **SheetError** con código estable, p. ej. **`EXTERNAL_VALIDATION_FAILED`**, y opcionalmente `params: { reason: 'network' | 'timeout' | 'server_error' }`, nivel `fatal` o `warning`. Así la UI puede mostrar un mensaje claro y la I18n traducir por código.
- **Abatibilidad:** El **AbortController** usado para cancelar la importación (ver **Architecture.md** — *Process Abort & Worker Cleanup*) debe pasarse al runner de table-validators para que cualquier **fetch** al backend pueda cancelarse cuando el usuario llame **`abort()`**. Si no, una petición pendiente podría seguir activa tras cancelar la importación.

El usuario registra sus validadores table (sync o async) con el mismo **Register()**; si son async y llaman al backend, la librería se encarga de try/catch y de pasar el AbortSignal.

### 6. Barras de progreso y throttling

Para evitar ruido visual y saturación de eventos:

- **Global:** Avance de la fase (0–100% basado en total de filas).
- **Local:** Avance del chunk actual.
- **Throttling:** El `EventTarget` en el Main Thread solo emitirá actualizaciones a la UI cada **16ms (60fps)** como máximo, independientemente de la velocidad de procesamiento del Worker.

## Web Worker y comunicación

- Toda la lógica de validación (iteración por chunks, celdas/filas y ejecución de validadores desde el Registry) corre dentro del **Web Worker**.
- Comunicación: **Comlink** para la API (expose/wrap). El worker expone un método tipo `validate(rawSheet, sheetLayout, options?, onProgress?)`. El **progreso** se reporta con un callback pasado con `Comlink.proxy(onProgress)`.
- **Retorno:** el worker **no** devuelve `Promise<sheet>`. Devuelve `Promise<ValidatorDelta>` (o similar) con `{ errors: Array<{ row, col?, code, level, params?, ... }> }`. El Main Thread aplica el **parche** sobre su copia de la sheet.
- El hilo principal no se bloquea. La UI de la barra de progreso se suscribe al **EventTarget** (`importer-progress`) con throttling de 16ms.
- **Abort y ciclo de vida:** El hook que instancia el Worker (p. ej. `useValidatorWorker`) debe guardar la instancia en una **ref** para que el Provider pueda llamar **`abort()`** (que ejecuta `worker.terminate()`); en el **cleanup de `useEffect`** al desmontar debe llamar **`worker.terminate()`**. Ver **Architecture.md** — *Process Abort & Worker Cleanup* y **2. Setting** — *Abort y ciclo de vida*.

## Validadores en el Worker (Registry)

- Las funciones validadoras **no** se envían por `postMessage`. El `sheetLayout` usa solo **identificadores string** (ej. `validators: ['required', 'is-email']`).
- El Worker importa un **ValidatorRegistry** que mapea cada string a **`{ fn, type: 'cell' | 'row' | 'table' }`**. El **Runner** usa `type` para saber cuándo invocar: **cell** y **row** en el bucle por filas; **table** una sola vez al final con el dataset completo. Así no hay que programar la lógica manualmente para cada importación. Ver **Architecture.md** — *Registry entry type*.
- Ubicación de implementaciones: p. ej. `src/utils/controller/[context]/` (ej. `cell-email-validator.ts`, `row-email-validator.ts`, `table-email-validator.ts`). Al registrar (desde el Provider o desde el Worker al cargar el layout), cada entrada debe incluir el **type** correcto (el **Register()** de cada controlador built-in lo aporta).

## Patrón Register() (built-in y custom)

Cada validador built-in (cell, row, sheet) **expone un helper Register()** para facilitar la construcción del layout sin magic strings y para registrar en el Worker si aún no lo está. Ver **Architecture.md** — *Register() pattern* y **2. Setting** — *Patrón Register()*.

- **Export por validador:** Ej. `CellEmailValidator` con `id: 'email'`, `validate: CellValidatorFn`, y `Register(registerValidator?: (name: string, fn: CellValidatorFn) => void): string`. `Register(registerValidator)` registra la función bajo `'email'` (si no estaba registrada) y devuelve `'email'`.
- **Uso en layout:** El consumidor puede escribir `validators: [CellEmailValidator.Register(registerValidator), 'phone']` — así no hay strings mágicos para los built-in y el Worker recibe la implementación al primer uso.
- **Interfaz para validadores custom:** La librería expone **`CellValidatorDefinition`** (y análogas `RowValidatorDefinition`, `SheetValidatorDefinition`) en `src/types/` para que el usuario cree sus propios validadores con el mismo contrato: `id`, `validate`, `Register(registerFn?)`. El usuario implementa `Register(registerFn) { if (registerFn) registerFn(this.id, this.validate); return this.id; }` y usa `MyValidator.Register(registerValidator)` en el layout.

**Funciones puras y params para configuración:** Si el registro al Worker usa serialización (`fn.toString()`), los validadores deben ser **puros** (no depender de variables externas/closures). Para casos como "el valor debe estar en una lista", la lista no debe vivir en un closure; debe pasarse por **params** en el layout: `validators: [{ name: 'in-list', params: { list: ['active', 'paused'] } }]`. La firma del validador incluye `(value, row, params?)`; el Worker invoca con los `params` que vienen en el layout (structured clone). Ver **Architecture.md** — *Technical note: Serialization to Worker and pure functions* y **2. Setting** — *Nota técnica: funciones puras y datos "externos"*.

## Plan de tareas (orden sugerido)

### Registry

1. Crear **ValidatorRegistry.ts** para mapear strings a **`{ fn, type: 'cell' | 'row' | 'table' }`**. Funciones de cell/row **síncronas**; las de table pueden ser **async**. El worker importa este Registry y resuelve los identificadores del layout.

### Runner

2. Implementar el **bucle de validación por chunks** (bloques de 100–500 filas): por cada chunk, ejecutar cell-validators (sync) y row-validators (sync); soportar **Early Exit** en caso de error fatal/crítico; acumular errores en una estructura de delta (row, col, SheetError). Luego implementar la **fase table**: módulo tipo **`run-table-validation.ts`** que ejecuta validadores con `type: 'table'` **una vez** con el dataset completo; si son async, usar **try/catch** y en fallo (red, 500, timeout) devolver **SheetError** con código **`EXTERNAL_VALIDATION_FAILED`**; recibir **AbortSignal** (del AbortController del proceso) y pasarlo al `fetch` para que **`abort()`** cancele la petición.

### Delta

3. Implementar en el Main Thread la función de **parcheo** que recibe el delta de errores del Worker y actualiza la copia local de la sheet (añadiendo errores a las celdas/filas/hoja correspondientes sin reemplazar la sheet entera).

### i18n

4. Asegurar que todos los validadores devuelvan objetos con **code** y **params** (y opcionalmente `message`), nunca solo strings, para que la UI pueda traducir por `code`.

### Worker

5. Configurar **validator.worker.ts** exponiendo la API de validación que devuelve solo el **delta de errores** (no la sheet). Incluir reporte de progreso vía callback (throttled en main).

### Hook y EventTarget

6. Implementar el hook (ej. **useValidatorWorker**) que instancia el worker, llama a `validate`, recibe el delta y aplica el parche; al recibir progreso, dispara el evento `importer-progress` en el EventTarget con **throttling de 16ms**.

### Tipos y mensajes

7. Definir tipos: **SheetError**, **ValidatorDelta** (array de errores con coordenadas), mensajes de progreso y contrato del evento `importer-progress`.

### Tests

8. Tests con **Vitest**: detección de errores **fatales** y comportamiento de **Early Exit**; que el Worker (o runner en test) no devuelva la sheet sino solo el delta; que el parche en Main Thread deje la sheet con los errores esperados; que el progreso y el throttling se comporten según lo definido.

## Criterios de aceptación

- [ ] Las funciones validadoras de **Celda** y **Fila** son **estrictamente síncronas** (MUST be synchronous for performance). Solo los validadores de **table** pueden ser async (backend/integridad).
- [ ] Validadores table async: **try/catch** en el runner; en fallo de red/backend devolver **SheetError** con código **`EXTERNAL_VALIDATION_FAILED`**; **AbortController** pasado para cancelar fetch al llamar **`abort()`**.
- [ ] El Worker **no devuelve la hoja** de datos, solo el **delta de errores**.
- [ ] El layout utiliza **identificadores tipo string** para los validadores (Registry en el worker).
- [ ] **Patrón Register():** Cada validador built-in (cell, row, sheet) exporta `id`, `type: 'cell'|'row'|'table'`, la función y `Register(registerFn?)` que devuelve el id y opcionalmente registra en el Worker **con el type** para que el Runner sepa cuándo invocar (cell/row por fila, table una vez al final). Interfaces **CellValidatorDefinition**, **RowValidatorDefinition**, **SheetValidatorDefinition** expuestas para validadores custom (ver Architecture y 2. Setting).
- [ ] **Table-level:** Validaciones con `type: 'table'` tienen acceso al dataset completo en el Worker; pueden producir errores multi-fila o globales de sheet; atomicidad con Transform (Transform solo inicia si table validations pasan o el usuario acepta continuar con errores).
- [ ] **Abort y cleanup:** El hook guarda el Worker en ref; cleanup en desmontaje llama `worker.terminate()`; el Provider puede terminar este Worker vía `abort()`.
- [ ] La UI se actualiza sin re-renders masivos mediante la suscripción al **EventTarget** con **throttling** (ej. 16ms).
- [ ] Los tests de Vitest cubren la detección de errores **fatales** y el **Early Exit**.
- [ ] Estructura de errores con **code** y **params** (i18n ready); parcheo del delta en Main Thread correcto.
