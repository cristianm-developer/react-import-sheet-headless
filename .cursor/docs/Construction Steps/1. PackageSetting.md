# 1. PackageSetting — Configuración del Package (react-import-sheet-headless)

Objetivo: configurar linting (ESLint), formato (Prettier), pipeline de calidad (Husky: commit-msg con Commitlint, pre-commit con lint-staged, pre-push con Vitest) y asegurar builds con tsup (CJS/ESM, minificado, tipos TS).

---

## Resumen de entregables

| Área            | Entregables |
|-----------------|-------------|
| ESLint          | `eslint.config.ts`, dependencias, ignorar `dist` y `node_modules` |
| Prettier        | `.prettierrc`, `.prettierignore` |
| Integración     | `eslint-config-prettier` en `eslint.config.ts` |
| Husky           | `.husky/`, script `prepare` en `package.json`, hooks `commit-msg`, `pre-commit` y `pre-push` |
| Commitlint      | Validación de mensajes de commit (Conventional Commits): `.commitlintrc.json`, hook `commit-msg` |
| lint-staged     | Sección `lint-staged` en `package.json`, hook `pre-commit` |
| Pre-push        | Hook `.husky/pre-push` que ejecuta `npm run test` (Vitest) |
| Cobertura       | Vitest coverage (v8), umbrales en `vitest.config.ts`; si la cobertura baja, `test` falla y el push se bloquea |
| Build           | Verificación/ajuste de `tsup.config.ts` y `package.json` exports |
| Commitizen (opcional) | Interfaz interactiva `git cz` para guiar a colaboradores al escribir commits |

## 1. ESLint

### 1.1 Instalación

```bash
npm install -D eslint @eslint/js typescript-eslint eslint-plugin-react-hooks eslint-plugin-react
```

### 1.2 Archivo de configuración

- **Archivo:** `eslint.config.ts` (raíz del package).
- **Ignorar:** `dist`, `node_modules`.
- **Bases:** `js.configs.recommended`, `tseslint.configs.recommended`.
- **Archivos:** todos los `*.ts` y `*.tsx`.
- **Plugins:** React, React Hooks.
- **Reglas:**

```ts
rules: {
  ...reactHooks.configs.recommended.rules,
  "react/jsx-uses-react": "off",
  "react/react-in-jsx-scope": "off",
  "@typescript-eslint/no-explicit-any": "warn",
  "no-console": ["warn", { allow: ["warn", "error"] }],
}
```

### 1.3 Script en `package.json`

- Añadir o reemplazar script `lint` para que ejecute ESLint (y opcionalmente `tsc --noEmit` si se desea comprobación de tipos en el mismo comando):

```json
"lint": "eslint ."
```

(O por ejemplo: `"lint": "eslint . && tsc --noEmit"` si se quiere incluir tipo-check.)

## 4. Husky y lint-staged (pipeline pre-commit)

### 4.1 Instalación

```bash
npm install -D prettier eslint-config-prettier husky lint-staged
```

### 4.2 Inicializar Husky

```bash
npx husky init
```

- Crea `.husky/` y suele añadir el script `prepare` en `package.json` (si no, añadirlo manualmente: `"prepare": "husky"`).

### 4.3 lint-staged en `package.json`

Añadir al final del `package.json` (o en la sección que se prefiera):

```json
"lint-staged": {
  "src/**/*.{ts,tsx,js,jsx}": [
    "eslint --fix",
    "prettier --write"
  ],
  "*.{json,md,css}": [
    "prettier --write"
  ]
}
```

### 4.4 Hook pre-commit

- **Archivo:** `.husky/pre-commit`.
- **Contenido:**

```bash
npx lint-staged
```

### 4.5 Flujo (pre-commit)

1. `git add .`
2. `git commit -m "mensaje"`
3. Husky ejecuta `pre-commit` → `npx lint-staged`.
4. lint-staged ejecuta ESLint (con `--fix`) y Prettier sobre los archivos en el commit.
5. Si ESLint falla (ej. error no auto-fixeable), el commit se aborta.
6. Si todo pasa, el commit se completa.

### 4.6 Hook pre-push (tests con Vitest)

El **pre-commit** cuida estilo y errores rápidos; el **pre-push** es el lugar ideal para ejecutar la suite de tests. Lógica: puedes trabajar en local con tests fallando, pero no subir al remoto si la lógica está rota.

**Crear el hook:**

Opción A (si tu versión de Husky lo soporta):

```bash
npx husky add .husky/pre-push "npm run test"
```

Opción B (si el comando anterior falla): crear manualmente el archivo `.husky/pre-push` con:

```bash
#!/usr/bin/env sh
. "$(dirname -- "$0")/_/husky.sh"

npm run test
```

### 4.7 Script de test en `package.json`

El script `test` debe ejecutarse en modo **run** (termina al acabar), no en modo **watch** (se quedaría colgado y el push nunca terminaría).

```json
"scripts": {
  "test": "vitest run",
  "test:watch": "vitest",
  "lint": "eslint .",
  "build": "tsup"
}
```

### 4.8 Validación de mensajes de commit (Commitlint – Conventional Commits)

Obliga a que los mensajes sigan un estándar (p. ej. `feat: add sheet parser`, `fix: validation logic`), lo que permite automatizar Changelog y versionado semántico, y ayuda a quien quiera participar a mantener un historial legible y profesional.

**Instalación:**

```bash
npm install -D @commitlint/config-conventional @commitlint/cli
```

**Configuración:** archivo `.commitlintrc.json` en la raíz:

```json
{
  "extends": ["@commitlint/config-conventional"]
}
```

Prefijos estándar habilitados: `feat:`, `fix:`, `docs:`, `style:`, `refactor:`, `test:`, `chore:` (entre otros).

**Hook de Husky (commit-msg):**

Opción A:

```bash
npx husky add .husky/commit-msg 'npx --no -- commitlint --edit ${1}'
```

Opción B (si falla): crear manualmente `.husky/commit-msg` con:

```bash
#!/usr/bin/env sh
. "$(dirname -- "$0")/_/husky.sh"

npx --no -- commitlint --edit $1
```

**Beneficios para colaboradores y proyecto:**

- Historial legible y consistente.
- Changelog automático futuro (Standard Version, Semantic Release).
- Versiones semánticas: `feat:` → minor (1.0.0 → 1.1.0), `fix:` → patch (1.0.0 → 1.0.1).

### 4.9 Commitizen (opcional – facilitar mensajes a colaboradores)

Para no tener que recordar los prefijos, se puede instalar Commitizen: interfaz interactiva en terminal que guía para armar el mensaje del commit.

```bash
npx commitizen init cz-conventional-changelog --save-dev --save-exact
```

Luego los colaboradores usan **`npm run commit`** (o `npx cz`) en lugar de `git commit -m "..."` y el asistente les pide tipo, alcance, descripción, etc. El script `commit` está configurado en `package.json`; el adapter es `cz-conventional-changelog` (config.commitizen.path).

### 4.10 Pipeline de seguridad total / “Escudo” de calidad

| Momento         | Herramienta     | Objetivo |
|-----------------|-----------------|----------|
| **Commit (mensaje)** | Commitlint       | Que el historial sea legible y profesional (Conventional Commits). |
| **Commit (código)**  | lint-staged      | Que no entre código sucio o mal formateado (ESLint + Prettier). |
| **Push**        | Vitest (pre-push) | Que no se suba lógica rota ni código nuevo sin tests (cobertura por debajo del umbral). |
| **Publish**     | script `publish` / tsup | Tests + lint + build; generar el bundle (CJS/ESM) para NPM. |

### 4.11 Ejemplo: push bloqueado por tests fallando

Si intentas hacer push con tests rotos, verás algo así:

```text
> vitest run
...
FAIL  src/hooks/useSheetImporter.test.ts > debe validar headers correctamente
AssertionError: expected false to be true

Error: [Husky] pre-push hook exited with code 1 (error)
error: failed to push some refs to 'github.com:usuario/mi-lib.git'
```

El push se cancela hasta que los tests pasen.

## 5. Cobertura de código (Vitest – Code Coverage)

Con el pre-push solo se asegura que los tests existentes no se rompan. Para **obligar a que cada nueva funcionalidad venga con sus propios tests** y mantener la calidad a largo plazo, hay que activar el check de cobertura. Si no, un colaborador podría subir una función nueva de 100 líneas sin un solo test; mientras no rompa los tests viejos, el pre-push lo dejaría pasar.

### 5.1 Instalar el motor de cobertura

Vitest tiene soporte nativo con v8 o istanbul:

```bash
npm install -D @vitest/coverage-v8
```

### 5.2 Umbrales de fallo (thresholds) en `vitest.config.ts`

Definir el mínimo de cobertura aceptable. Si el código nuevo baja de ese porcentaje, `npm run test` falla y, por tanto, el **pre-push** también.

```ts
// vitest.config.ts
import { defineConfig } from 'vitest/config';

export default defineConfig({
  test: {
    environment: 'jsdom',
    coverage: {
      provider: 'v8',
      reporter: ['text', 'json', 'html'],
      thresholds: {
        lines: 80,       // Si el total de líneas cubiertas es < 80%, falla
        functions: 80,
        branches: 70,    // Útil para validar que todos los if/else se prueben
        statements: 80,
      },
    },
  },
});
```

La carpeta `coverage` ya está en `.prettierignore`; conviene añadirla también a `.gitignore` si no está.

### 5.3 Cómo obliga esto al colaborador

1. El colaborador hace `git push`.
2. Husky ejecuta `npm run test`.
3. Vitest corre los tests y calcula qué líneas/funciones/ramas se ejecutaron.
4. Si añadió código nuevo (ej. una función `validateCuit`) sin tests, la cobertura total bajará del 80%.
5. Vitest lanza error, el proceso termina con código distinto de 0 y el push se detiene, con un mensaje indicando que no se alcanza el umbral de cobertura requerido.

### 5.4 Niveles recomendados para una librería NPM

| Métrica     | Rango sugerido | Motivo |
|-------------|----------------|--------|
| **Lines**   | 80–90 %        | Asegura que casi todo el código se ejecute al menos una vez. |
| **Branches**| 70–80 %        | Crítico para validadores (sheets, etc.): obliga a testear tanto el caso válido como el de error. |
| **Functions** | 80 %         | Cada función tiene al menos un test. |
| **Statements** | 80 %      | Alineado con líneas. |

Empezar con los valores del ejemplo (80/80/70/80) y subir si la base de tests ya es sólida.

## 6. Build con tsup (TS, JS, CJS, ESM, minificado)

### 6.1 Estado actual de `tsup.config.ts`

- `entry: ['src/index.ts']`
- `format: ['cjs', 'esm']`
- `dts: true` (genera `.d.ts`)
- `minify: true`
- `external: ['react']` (y conviene tener `react-dom` si se usa)
- `sourcemap: true`

Objetivo: librería compatible en TS/JS, con salida CommonJS y ESM y minificado. La configuración actual ya cumple; solo verificar que los nombres de salida coincidan con `package.json`.

### 6.2 Coherencia con `package.json` exports

- **Import (ESM):** debe apuntar al artefacto ESM (p. ej. `./dist/index.js` si tsup genera ESM como `.js`).
- **Require (CJS):** debe apuntar al artefacto CJS (p. ej. `./dist/index.cjs`).
- **Tipos:** `types` → `./dist/index.d.ts`.

En tsup con `format: ['cjs', 'esm']` la salida típica es:

- ESM: `dist/index.js`
- CJS: `dist/index.cjs`
- Tipos: `dist/index.d.ts`

Comprobar que en `package.json`:

- `"import"` → `"./dist/index.js"`
- `"require"` → `"./dist/index.cjs"`
- `"types"` → `"./dist/index.d.ts"`

Si en tu versión de tsup los nombres fueran distintos (p. ej. `.mjs` para ESM), ajustar `exports` y, si hace falta, opciones de tsup para que los nombres coincidan.

### 6.3 Script de build

- Mantener o usar: `"build": "tsup"` (o `"build": "tsup --minify"` si se quiere ser explícito; en el plan ya está `minify: true` en la config).

## Orden recomendado de implementación

1. **ESLint:** instalar deps, crear `eslint.config.ts` (sin Prettier aún), ignorar `dist` y `node_modules`, añadir script `lint`.
2. **Prettier:** instalar Prettier y `eslint-config-prettier`, crear `.prettierrc` y `.prettierignore`.
3. **Integración:** añadir `eslint-config-prettier` al final de `eslint.config.ts`.
4. **Husky + lint-staged:** instalar, `npx husky init`, configurar `lint-staged` en `package.json`, contenido de `.husky/pre-commit` y **pre-push** (`.husky/pre-push` con `npm run test`).
5. **Commitlint:** instalar `@commitlint/cli` y `@commitlint/config-conventional`, crear `.commitlintrc.json`, añadir hook `.husky/commit-msg` con `commitlint --edit $1`.
6. **Script test:** asegurar `"test": "vitest run"` y `"test:watch": "vitest"` en `package.json`.
7. **Cobertura Vitest:** instalar `@vitest/coverage-v8`, configurar en `vitest.config.ts` `coverage.provider: 'v8'`, `reporter` y `thresholds` (lines, functions, branches, statements) para que el test falle si se baja del mínimo.
8. **tsup:** revisar que `tsup.config.ts` y `package.json` exports estén alineados (CJS/ESM, minify, tipos).
9. **Commitizen (opcional):** si se quiere guiar a colaboradores, `npx commitizen init cz-conventional-changelog --save-dev --save-exact`.

## Uso de Commitizen (colaboradores)

Para crear commits con mensaje Conventional sin memorizar prefijos:

- **`npm run commit`** — lanza la interfaz de Commitizen (tipo, alcance, descripción, etc.).
- **`npx cz`** — equivalente si no se usa el script.

El mensaje generado cumple con Commitlint (`@commitlint/config-conventional`). Alternativa manual: `git commit -m "feat: add X"` (prefijos: feat, fix, docs, style, refactor, test, chore).
